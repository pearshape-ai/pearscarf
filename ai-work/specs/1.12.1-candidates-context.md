# Entity resolution — candidate context package

## Context

- 1.12.0 broadened retrieval to return a list of candidate entities
- The LLM judge (1.12.2) needs more than just name and metadata to make confident decisions — it needs to know what each candidate is connected to in the graph
- A bare candidate like `{name: "David Kim", email: "david@pearventures.io"}` is insufficient to distinguish between two Davids; knowing David Kim manages the Meridian API Integration makes the match obvious
- This spec builds the context package that gets passed to the resolution prompt

## TODO

### `graph.py` — new function `get_entity_context()`

- Accepts an entity ID
- Returns a dict with:
  - `entity` — name, type, metadata (email, role, domain)
  - `facts` — list of current (non-invalidated) fact edges on this entity, each with `category`, `fact` text, `other_name` (the connected entity name or Day date), `confidence`
  - `connections` — list of directly connected entity names and their relationship category (1-hop traversal, current edges only)
- Reuses `get_facts_for_entity()` and `traverse_fact_edges()` already in `graph.py` — no new Cypher needed
- Limits facts to 10 most recent by `valid_at` to keep the context package token-efficient
- Limits connections to 10 to avoid bloating the LLM input

### `indexer.py` — build context package per candidate

- After `find_entity_candidates()` returns a non-empty list, call `get_entity_context()` for each candidate
- Assemble a list of context packages — one per candidate — ready to be passed to the resolution prompt
- Store the assembled list on the resolution state for the current entity — 1.12.3 will consume it
- For now (pre-1.12.3), log the context package at debug level and continue with the existing temporary fallback

## What NOT to do

- No LLM calls — this spec is context assembly only
- No changes to retrieval logic from 1.12.0
- No resolution decisions — that is 1.12.3
- Do not fetch historical (invalidated) facts — current facts only, the judge needs current state
- Do not traverse more than 1 hop — deeper traversal adds noise without improving resolution accuracy
- No changes to the extraction prompt
