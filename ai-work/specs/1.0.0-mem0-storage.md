# Context

- Current storage: SQLite facts table (poor quality output), SQLite graph (entities, edges), ChromaDB (vectors).
- The facts table doesn't produce useful context. Extraction quality is poor, dedup is fragile, and the stored data doesn't help the worker make better decisions.
- Three separate storage systems doing overlapping work with hand-rolled logic for extraction, dedup, and conflict resolution.
- Mem0 handles the hard parts — extraction, deduplication, conflict resolution, and retrieval — through LLM-driven pipelines.
- Neo4j chosen as graph backend because it's first-class in both Mem0 and Graphiti. Same instance reused when Graphiti is integrated later.

## Pluggable Memory Backend

Memory backend configured via env var (e.g. MEMORY_BACKEND=mem0). The Retriever checks this config and calls the right backend.

When not configured, falls back to the existing SQLite facts + ChromaDB pipeline. This keeps the system runnable without Mem0 during development or testing. Same pattern will be used for Graphiti and Cognee later.

## Indexer Changes

Current Indexer does heavy lifting: builds extraction prompt, calls LLM, parses JSON, resolves entities, creates edges, upserts facts, embeds in ChromaDB.

With Mem0, the Indexer passes email content plus any human context from triage to Mem0's add. Mem0's internal LLM handles extraction, entity resolution, graph population, and embedding. The Indexer shrinks significantly.

## Retriever Changes

Current Retriever runs three queries (facts lookup, graph traversal, vector search) and assembles a context package.

With Mem0, the Retriever makes one search call. Response contains vector-matched memories and graph-connected entities. The Retriever packages these for the worker in the same context package format — the worker doesn't know the backend changed.

## What Mem0 Replaces

- Facts table → Mem0 memories with LLM-driven dedup and conflict resolution
- SQLite graph (entities, edges) → Neo4j graph via Mem0's graph memory
- ChromaDB → Mem0's vector layer

## What Stays in SQLite

- Sessions + messages (agent communication)
- Records, emails, issues (System of Record, expert-owned)
- Discord thread mappings
- Classification fields on records (triage still owned by worker)

Raw data is always available for re-processing, backfilling, or switching to a different memory backend.

## Expected Outcomes

For an email from Michael at Acme about a Q3 invoice, Mem0 should extract: Michael is a person who works at Acme Corp, Acme Corp is a company, the invoice amount, and the relationship between them. When the worker later asks "what do I know about Acme Corp?", the search should return memories about Acme, connected people like Michael, and their relationships.

For human context during triage ("Michael is a potential partner for API integration"), Mem0 should store this as a high-signal memory associated with both Michael and Acme, surfacing it in future queries about either.

For contradictions ("invoice was $3k" then later "invoice is $5k"), Mem0's conflict resolution should update rather than duplicate. This is the specific problem the facts table couldn't handle.
