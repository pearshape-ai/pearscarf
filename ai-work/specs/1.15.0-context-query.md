# context_query.py: single data access layer and retriever rewire

## Context

- `retriever.py` currently calls `graph.py` and `vectorstore.py` directly from within each tool's `execute()` method
- The MCP server (1.15.1) will need to expose the same data — building a separate query path would create two diverging retrieval surfaces
- The correct design: a single module `context_query.py` that is the only layer responsible for translating data asks into data. Both `retriever.py` and `mcp_server.py` call through it.
- `context_query.py` is read-only and storage-agnostic — it calls graph.py, store.py, and vectorstore.py as needed

## Naming note

`graph.get_entity_context()` already exists — it is the resolution judge's context package used during entity disambiguation. It is not the same as the MCP tool `get_entity_context`, which is a higher-level composition built in 1.15.4. Do not rename or touch the existing `graph.get_entity_context()`. The `context_query.py` module does not expose a function by that name in this spec.

## TODO

### 1. Create `context_query.py`

New module at `pearscarf/context_query.py`. All functions are read-only. None write to any storage.

#### `find_entity(name, entity_type=None) → list[dict]`

Wraps `graph.search_entities(name, entity_type=entity_type, limit=10)`.

Returns list of entity dicts: `{id, name, type, metadata}`.

Used as the name resolution step before any entity-scoped query.

---

#### `get_facts(entity_id, edge_label=None, fact_type=None, include_stale=False, since=None) → list[dict]`

Wraps `graph.get_facts_for_entity(entity_id, include_stale=include_stale)`.

Post-filters in Python:
- `edge_label`: keep only facts where `fact["edge_label"] == edge_label`
- `fact_type`: keep only facts where `fact["fact_type"] == fact_type`
- `since`: keep only facts where `fact["source_at"] >= since` (ISO string comparison)

Returns list using the canonical fact object shape (see architecture doc).

This is the workhorse. `get_transitions`, `get_open_items` in later specs are compositions of this function — not new functions. They are convenience wrappers with preset `edge_label` and `fact_type` parameters.

---

#### `get_connections(entity_id, max_depth=1, include_stale=False, edge_labels=None) → dict`

Wraps `graph.traverse_fact_edges(entity_id, max_depth=max_depth, current_only=not include_stale, edge_labels=edge_labels)`.

Returns the traversal dict as-is: `{nodes, edges, source_records}`.

---

#### `get_facts_for_day(date) → list[dict]`

Wraps `graph.get_facts_for_day(date)`.

Returns list of fact dicts with entity context included.

---

#### `get_path(entity_id_a, entity_id_b) → dict`

New Cypher query — does not exist in `graph.py` yet. Add `get_path(entity_id_a, entity_id_b)` to `graph.py` first, then call it here.

The Cypher query: shortest undirected path between the two nodes across fact-edges. Current edges only (`stale=false`). Max depth 4 to avoid expensive traversals.

Returns: `{path: [{entity, edge_label, fact_type, fact, direction}], direct_facts: [{edge_label, fact_type, fact, source_at}]}`

`direct_facts` are edges where both endpoints are the two queried entities directly (depth 1). `path` is the shortest traversal sequence when no direct edge exists.

---

#### `get_conflicts(entity_id=None) → list[dict]`

New Cypher query — does not exist in `graph.py` yet. Add `get_conflicts(entity_id=None)` to `graph.py` first, then call it here.

The Cypher query: find AFFILIATED fact-edges where two current (stale=false) edges exist for the same `(from_id, fact_type, to_id)` slot. If `entity_id` is provided, scope to that entity as `from_id`.

Returns: `[{entity_name, edge_label, fact_type, fact_a, fact_b, source_at_a, source_at_b, edge_id_a, edge_id_b}]`

---

#### `get_communications(entity_id, since=None) → list[dict]`

New Postgres query — does not exist in `store.py` yet. Add `get_communications_for_entity(entity_id, since=None)` to `store.py` first, then call it here.

Resolves `entity_id` to an entity name/email via `graph.get_entity()`, then queries the emails table for records where the entity appears as sender or recipient.

```sql
SELECT e.record_id, e.sender, e.recipient, e.subject,
       e.received_at, r.source
FROM emails e
JOIN records r ON e.record_id = r.id
WHERE (e.sender ILIKE %name_or_email%
   OR e.recipient ILIKE %name_or_email%)
  AND (since IS NULL OR e.received_at >= since)
ORDER BY e.received_at DESC
LIMIT 20
```

Returns: `[{record_id, sender, recipient, subject, received_at, source}]`

---

#### `vector_search(query, n_results=5) → list[dict]`

Wraps `vectorstore.query(query, n_results=n_results)`.

Returns results as-is: `[{id, content, metadata, score}]`.

---

### 2. Add new functions to `graph.py`

Two new read-only functions needed by `context_query.py`:

**`get_path(entity_id_a, entity_id_b, max_depth=4) → dict`**

Cypher: `MATCH path = shortestPath((a)-[*1..{max_depth}]-(b))` where both nodes are matched by element ID and all edges in path have `stale=false`. Returns `{path, direct_facts}` as described above.

**`get_conflicts(entity_id=None) → list[dict]`**

Cypher: match pairs of AFFILIATED edges on the same `(from, fact_type, to)` slot where both have `stale=false` and `replaced_by IS NULL`. Optionally scoped to a specific `from_id`. Returns list as described above.

### 3. Add new function to `store.py`

**`get_communications_for_entity(name_or_email, since=None) → list[dict]`**

Postgres query against the emails table as described in step 1. `name_or_email` is a string for ILIKE matching.

### 4. Rewire `retriever.py`

Update each tool's `execute()` to call `context_query` instead of `graph` or `vectorstore` directly. No changes to tool names, descriptions, or input schemas — only the `execute()` body changes.

| Tool | Current call | New call |
|---|---|---|
| `SearchEntitiesTool.execute()` | `graph.search_entities(query, entity_type, limit=5)` | `context_query.find_entity(query, entity_type)` |
| `FactsLookupTool.execute()` | `graph.get_facts_for_entity(entity_id, include_stale)` | `context_query.get_facts(entity_id, include_stale=include_stale)` |
| `GraphTraverseTool.execute()` | `graph.traverse_fact_edges(entity_id, max_depth, current_only, edge_labels)` | `context_query.get_connections(entity_id, max_depth, include_stale, edge_labels)` |
| `DayLookupTool.execute()` | `graph.get_facts_for_day(date_str)` | `context_query.get_facts_for_day(date_str)` |
| `VectorSearchTool.execute()` | `vectorstore.query(query, n_results)` | `context_query.vector_search(query, n_results)` |

After this change, `retriever.py` has no direct imports of `graph` or `vectorstore`.

### 5. Update imports

- `retriever.py`: remove `from pearscarf import graph, vectorstore`, add `from pearscarf import context_query`
- `context_query.py`: imports `from pearscarf import graph, vectorstore` and `from pearscarf.store import get_communications_for_entity`

## What NOT to do

- Do not implement MCP wiring here — that is 1.15.1
- Do not change tool names, descriptions, or input schemas in `retriever.py` — only the `execute()` bodies change
- Do not touch `graph.get_entity_context()` — it is the resolution judge's function, not an MCP tool
- Do not implement `get_entity_context` as an MCP-facing function here — that is 1.15.4
- Do not implement `get_open_items`, `get_transitions` as separate functions — they are compositions of `get_facts` with preset parameters, implemented in 1.15.4 and 1.15.5
- Do not move any write functions — `context_query.py` is read-only throughout
- Do not change `graph.py` write functions (create_fact_edge, mark_fact_stale, etc.) — only two new read functions are added
