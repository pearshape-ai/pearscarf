### 1. Register `pearscarf.get_relationship`

```python
@mcp.tool(description=(
    "Find how two entities are connected in the PearScarf graph. "
    "Returns direct facts between them and the shortest path if no direct connection exists. "
    "Use before drafting a message or making a decision involving two parties."
))
def get_relationship(entity_a: str, entity_b: str) -> dict:
```

Parameters:
- `entity_a` — surface form of the first entity (required). Resolved internally.
- `entity_b` — surface form of the second entity (required). Resolved internally.

Resolution: resolve both names via `find_entity`. If either fails to resolve, return:
```json
{"error": "not_found", "name": "<the name that failed>"}
```

If both resolve, call `context_query.get_path(entity_id_a, entity_id_b)`.

Returns:
```json
{
  "entity_a": {"id": "...", "name": "Marcus Webb", "type": "person"},
  "entity_b": {"id": "...", "name": "Meridian Deal", "type": "project"},
  "direct_facts": [<canonical fact objects>],
  "path": [
    {"entity": "Marcus Webb", "edge_label": "ASSERTED", "fact_type": "commitment", "fact": "...", "direction": "outbound"},
    ...
  ]
}
```

No path between the entities is a valid answer — not an error:
```json
{
  "entity_a": {...},
  "entity_b": {...},
  "direct_facts": [],
  "path": []
}
```

`direct_facts` are populated when entities are directly connected (depth 1). `path` shows the traversal sequence for indirect connections. If direct facts exist, `path` is still populated with the same single-hop edges — agents can use whichever is more useful.

### 2. Register `pearscarf.get_conflicts`

```python
@mcp.tool(description=(
    "Find AFFILIATED facts where the graph holds two current conflicting values for the same slot. "
    "These are cases where the Curator detected equal source_at timestamps and could not resolve automatically. "
    "Use when reviewing graph health or validating entity affiliations."
))
def get_conflicts(entity_name: str = None) -> dict:
```

Parameters:
- `entity_name` — optional surface form to scope to one entity. If omitted, returns all conflicts in the graph.

If `entity_name` is provided: resolve via `find_entity`. If not found, return `{"error": "not_found", "name": entity_name}`.

If `entity_name` is omitted: call `context_query.get_conflicts()` globally — no entity resolution needed.

Returns:
```json
{
  "conflicts": [
    {
      "entity_name": "James Whitfield",
      "edge_label": "AFFILIATED",
      "fact_type": "employee",
      "fact_a": "VP of Product at Meridian",
      "fact_b": "VP of Engineering at Meridian",
      "source_at_a": "2026-03-05T09:00:00Z",
      "source_at_b": "2026-03-05T09:00:00Z",
      "edge_id_a": "...",
      "edge_id_b": "..."
    }
  ],
  "count": 1
}
```

No conflicts is not an error — return `{"conflicts": [], "count": 0}`.

## What NOT to do

- Do not add max_depth or include_stale parameters to `get_relationship` — keep it simple. Agents that need deeper traversal use `get_connections`.
- Do not return an error when no path exists — empty `path` and `direct_facts` is the correct response
- Do not scope `get_conflicts` to ASSERTED or TRANSITIONED — conflicts are AFFILIATED only, by design
- Do not implement `get_entity_context` here — that is 1.15.4
