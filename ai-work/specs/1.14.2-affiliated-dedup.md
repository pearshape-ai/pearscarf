# Curator: AFFILIATED semantic dedup and confidence upgrade

## Context

- The write loop always writes AFFILIATED facts unless they are literal dups — semantically equivalent AFFILIATED edges from different records accumulate in the graph
- A person can hold multiple distinct roles at the same entity (e.g. Head of Operations and Head of Partnerships) — these must coexist
- But the same role described differently across records (e.g. "VP Eng at Acme" and "VP of Engineering at Acme Corp") is a duplicate — only the most recent should be current
- This spec also introduces `curator_judge.py` — the reusable LLM judge module used here and in 1.14.3

### 1. Create `curator_judge.py`

A module with a single function:

```
judge_equivalence(candidates, edge_label) → groups
```

**Input:**
- `candidates` — list of fact dicts, each with: `edge_id`, `fact`, `role`, `source_at`, `confidence`, `source_record`
- `edge_label` — `"AFFILIATED"` or `"ASSERTED"` — controls the prompt variant

**Output:**
- `groups` — list of lists. Each inner list contains `edge_id` values that are semantically equivalent. Edge IDs that appear in their own group of one are genuinely distinct — no action needed.

**One call per slot.** The function receives all candidates for one `(from_entity_id, fact_type, to_entity_id)` slot and returns groups for that slot in a single LLM call.

**Prompt variant for AFFILIATED:**
- Ask the judge to group the candidates by whether they describe the same real-world role or relationship
- Genuinely distinct roles (e.g. two different jobs at the same company) must not be grouped together
- Role string, fact text, and source context are all available to the judge

**Prompt variant for ASSERTED (for 1.14.3):**
- Placeholder for now — not used in this spec. 1.14.3 will fill in the ASSERTED prompt variant.

**Response format:** prompt the judge to return JSON only — a list of lists of edge IDs. Parse and return directly. If parsing fails, log a warning and return each candidate in its own group (treat as distinct — safe fallback, no staleness applied).

### 2. Add AFFILIATED dedup to `curator.py` processing function

Replace the stub in `curator.py`'s processing function with the AFFILIATED dedup pass:

**Step 1 — fetch this record's AFFILIATED edges:**
Call `get_nodes_by_source_record(record_id)` and filter for `edge_label == "AFFILIATED"`.

If none: return early — nothing to do.

**Step 2 — for each unique `(from_entity_id, fact_type, to_entity_id)` slot in this record's edges:**

Query the graph for all current (non-stale) AFFILIATED edges for this slot across all records — including the one just written.

If only one edge exists for the slot: skip — nothing to compare.

**Step 3 — call the judge:**

Pass all candidates for this slot to `curator_judge.judge_equivalence(candidates, "AFFILIATED")`.

**Step 4 — process groups:**

For each group returned by the judge:
- Group of one: genuinely distinct — skip
- Group of two or more: semantically equivalent — stale all but the one with the most recent `source_at`
  - Most recent `source_at` is the survivor
  - Call `mark_fact_stale(older_edge_id, survivor_edge_id)` for each non-survivor
  - Equal `source_at` within a group: stale none — log as unresolved, move on

**Step 5 — confidence upgrade:**

After all groups for this slot are processed, check the surviving edge's `source_records` list. Fetch the `confidence` value of the fact as produced by each source record.

- If any source record produced the fact as `stated`: set surviving edge `confidence = stated`
- If all produced it as `inferred`: leave as `inferred`
- If surviving edge already has `confidence = verified` or `confidence = retracted`: do not change

Log any upgrade: `"confidence upgraded: {edge_id} inferred → stated (source: {record_id})"`

### 3. Logging

One log entry per outcome per slot:

- `"affiliated dedup: slot ({from}, {fact_type}, {to}) — {n} candidates, {k} groups"`
- `"affiliated staled: {older_id} → {survivor_id} (source_at: {older} < {survivor})"`
- `"affiliated unresolved: equal source_at for equivalent edges {id_a}, {id_b} — skipped"`
- `"affiliated distinct: {n} genuinely distinct roles at same slot — all kept"`
- `"confidence upgraded: {edge_id} inferred → stated (source: {record_id})"`
- `"curator_judge parse error: slot ({from}, {fact_type}, {to}) — treating all as distinct"`

## What NOT to do

- Do not batch multiple slots into a single LLM call — one call per slot
- Do not filter candidates by role before calling the judge — the judge decides what is equivalent
- Do not implement the ASSERTED prompt variant in `curator_judge.py` — that is 1.14.3
- Do not implement expired commitment detection — that is 1.14.4
- Do not touch `graph.py` — `mark_fact_stale` and `get_nodes_by_source_record` are already in place
- Do not stale edges when `source_at` values are equal — log as unresolved and leave both current
