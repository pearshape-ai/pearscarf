# curator_queue table and indexer enqueue on indexed=true

## Context

- The indexer processes records one at a time and marks each `indexed=true` via `_mark_indexed(record_id)` after all facts are written
- The curator needs to know which records are ready to process ‚Äî currently there is no signal, no queue, and no trigger
- The queue must live in Postgres ‚Äî same pattern as `resolution_pending`, visible and debuggable, no external message broker needed
- The indexer must never be blocked by queue failures ‚Äî enqueuing is best-effort

## TODO

### 1. Create `curator_queue` table

Add a new migration to create the table with the following columns:

| Column | Type | Notes |
|---|---|---|
| `record_id` | text | Primary key. Same value as `records.id` |
| `queued_at` | timestamptz | Set to `now()` on insert |
| `claimed_at` | timestamptz | Null when unclaimed. Set by the runner when claimed. |

Constraints:
- `record_id` is the primary key ‚Äî enforces uniqueness, re-indexing the same record does not create duplicate queue entries
- `claimed_at` defaults to null

No foreign key constraint on `record_id` ‚Äî the queue is operationally decoupled from the records table. A queue entry for a deleted record is harmless and will be skipped by the runner.

### 2. Add `_enqueue_for_curation(record_id)` to `store.py`

New function, called by the indexer after `_mark_indexed` succeeds:

- Insert `(record_id, now(), null)` into `curator_queue`
- On conflict on `record_id`: do nothing ‚Äî idempotent, re-indexing does not reset `claimed_at`
- Returns nothing ‚Äî caller does not need to know the result

### 3. Call `_enqueue_for_curation` from `indexer._process_record`

After `_mark_indexed(record_id)` succeeds, call `_enqueue_for_curation(record_id)`.

Wrap the call in a try/except:
- On any exception: log a warning with `record_id` and the error, continue
- The indexer must not raise or stop processing due to a queue failure

### 4. Add `psc queue` CLI command

Extend `cli.py` with a `queue` subcommand for inspecting the curator queue:

- `psc queue` ‚Äî shows count of unclaimed entries, count of claimed entries, oldest `queued_at`
- `psc queue list` ‚Äî lists up to 20 entries with `record_id`, `queued_at`, `claimed_at`
- `psc queue clear` ‚Äî deletes all unclaimed entries (for dev/reset use only). Requires `--confirm` flag.

## What NOT to do

- Do not implement the runner or any agent logic here ‚Äî that is 1.14.1
- Do not add a foreign key from `curator_queue.record_id` to `records.id` ‚Äî operational decoupling is intentional
- Do not block or raise in the indexer if enqueuing fails ‚Äî best-effort only
- Do not reset `claimed_at` on conflict ‚Äî a re-indexed record that is already claimed should not interrupt an in-progress agent run
