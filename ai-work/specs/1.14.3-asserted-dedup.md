# Curator: ASSERTED semantic dedup and confidence upgrade

## Context

- ASSERTED facts always accumulate in the write loop — the same claim worded differently across records produces multiple current edges for the same slot
- Unlike AFFILIATED, multiple ASSERTED facts between the same two entities can legitimately coexist — two distinct commitments are both valid. Only semantically equivalent ones (same thing, different wording) should be collapsed.
- `curator_judge.py` already exists from 1.14.2 with `judge_equivalence(candidates, edge_label)`. This spec adds the ASSERTED prompt variant and runs dedup for all ASSERTED fact_types.
- This spec also fixes the source record query for both AFFILIATED and ASSERTED: `$rid IN r.source_records` instead of `source_record = $rid`. This ensures edges whose `source_records` list was extended by a literal dup merge are also visible to the Curator.

### 2. Add ASSERTED prompt variant to `curator_judge.py`

Add the ASSERTED branch to `judge_equivalence`. When `edge_label == "ASSERTED"`:

- Ask the judge to group candidates that express the same underlying claim — same obligation, same decision, same concern — even if worded differently
- Emphasize that distinct claims must NOT be grouped together. Two commitments to the same entity about different things are both valid and must survive.
- The judge has access to: `fact` text, `fact_type`, `source_at`, `confidence`, `source_record` for each candidate
- Same response format as AFFILIATED: a list of lists of `edge_id` values
- Same parse failure fallback: treat all as distinct — safe, no staleness applied

The ASSERTED prompt is harder than AFFILIATED. The equivalence bar is higher — false positives (wrongly collapsing distinct claims) are worse than false negatives (missing a duplicate). The prompt should make this explicit.

### 3. Add ASSERTED dedup pass to `curator.py`

After the AFFILIATED dedup pass, run a second pass for ASSERTED edges from the same record. The structure is identical to the AFFILIATED pass:

**Step 1 — fetch this record's ASSERTED edges:**
Filter edges returned by `get_edges_by_source_record(record_id)` for `edge_label == "ASSERTED"`.

All `fact_type` values are included — no filtering by fact_type. Every recorded claim should be as clean as possible.

If none: skip.

**Step 2 — for each unique `(from_entity_id, fact_type, to_entity_id)` slot:**

Query the graph for all current (non-stale) ASSERTED edges for this slot across all records.

If only one edge exists for the slot: skip.

**Step 3 — call the judge:**

Pass all candidates to `curator_judge.judge_equivalence(candidates, "ASSERTED")`.

**Step 4 — process groups:**

Same staleness logic as AFFILIATED:
- Group of one: genuinely distinct — skip
- Group of two or more: semantically equivalent — stale all but the one with the most recent `source_at` via `mark_fact_stale(older_id, survivor_id)`
- Equal `source_at` within a group: stale none — log as unresolved

**Step 5 — confidence upgrade:**

After all groups for this slot are processed, check the surviving edge's `source_records` list:
- If any source record produced the fact as `stated`: set surviving edge `confidence = stated`
- If all produced it as `inferred`: leave as `inferred`
- If the surviving edge already has `confidence = verified` or `confidence = retracted`: do not change

Log any upgrade: `"confidence upgraded: {edge_id} inferred → stated (source: {record_id})"`

### 4. Logging

Same pattern as AFFILIATED, prefixed with `asserted`:

- `"asserted dedup: slot ({from}, {fact_type}, {to}) — {n} candidates, {k} groups"`
- `"asserted staled: {older_id} → {survivor_id} (source_at: {older} < {survivor})"`
- `"asserted unresolved: equal source_at for equivalent edges {id_a}, {id_b} — skipped"`
- `"asserted distinct: {n} genuinely distinct claims at same slot — all kept"`
- `"confidence upgraded: {edge_id} inferred → stated (source: {record_id})"`
- `"curator_judge parse error: slot ({from}, {fact_type}, {to}) — treating all as distinct"`

## What NOT to do

- Do not filter by `fact_type` before running dedup — all ASSERTED fact_types are checked
- Do not batch multiple slots into a single LLM call — one call per slot
- Do not set a lower equivalence bar than AFFILIATED — false positives on ASSERTED (wrongly collapsing distinct claims) are worse than false negatives
- Do not implement expired commitment detection here — that is 1.14.4
- Do not stale edges when `source_at` values are equal — log as unresolved and leave both current
- Do not remove the AFFILIATED pass from `curator.py` — both passes run on every record, in order: AFFILIATED first, then ASSERTED
