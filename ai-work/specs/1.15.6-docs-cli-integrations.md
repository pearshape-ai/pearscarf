**Data flow — read path:**
- retriever.py (internal agents) → context_query.py → graph.py / store.py / vectorstore.py

- Neo4j (graph db) — entities and fact edges
- Postgres — records, emails, issues, curator_queue, mcp_keys

**Key labels:**
- Write path nodes: Indexer, Curator
- Read path nodes: Retriever (internal), MCP Server (external)
- Shared layer: context_query.py
- Annotation: "writes" on indexer/curator → storage arrows, "reads only" on context_query → storage arrows

Generate the diagram as an SVG file at `docs/architecture.svg`. Reference it in `docs/architecture.md` with `![PearScarf Architecture](architecture.svg)`.

The `docs/architecture.md` prose section beneath the diagram explains the separation in one paragraph: indexer and curator own all writes; context_query is the only read layer for context-building; retriever and MCP are two surfaces over the same functions — internal agents reason about what to call, external agents call directly.

### 5. Add `psc query` CLI command

```
psc query <tool_name> [--param value ...]
```

Calls the named tool via `context_query.py` directly (not via MCP — no auth needed for local dev). Prints the JSON response.

Examples:
```
psc query find_entity --name "Marcus Webb"
psc query get_facts --entity-name "Marcus Webb" --edge-label AFFILIATED
psc query get_open_commitments
psc query get_recent_activity --entity-name "Meridian Deal" --since 2026-03-01
```

All 8 tools supported. Parameters follow the same names as the MCP tool parameters, using `--kebab-case` for CLI convention.

### 6. Integration test pass

Add `psc eval --integration` mode (or a standalone `psc integration-test` command) that:

1. Seeds the base eval dataset
2. Calls all 8 tools against known entities from the dataset
3. Validates response shapes — correct structure, no missing required fields, no raw Neo4j objects
4. Validates response content — spot checks against expected values from the eval ground truth
5. Prints pass/fail per tool with a summary

This is not a comprehensive eval — it's a sanity check that the full stack works end-to-end. The comprehensive eval for retrieval quality is a separate concern.

## What NOT to do

- Do not write API reference docs for internal functions in `graph.py` or `store.py` — `docs/context_query.md` is the public interface
- Do not implement the `psc query` command as an MCP client — it calls `context_query.py` directly, no auth token needed
- Do not make the integration test a substitute for the full eval pipeline — it's a smoke test, not a scorer
