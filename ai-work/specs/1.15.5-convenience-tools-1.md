# Convenience tools: get_open_commitments, get_open_blockers, get_recent_activity

## Context

- All convenience foundation tools are live (1.15.4)
- `get_open_commitments` and `get_open_blockers` are `get_facts` with preset filters
- `get_recent_activity` is the most complex convenience tool — it hits two storage layers: graph (TRANSITIONED + ASSERTED[reference]) and Postgres records table (communication metadata)
- `context_query.get_communications` exists for the Postgres side (1.15.0)

### 1. Register `pearscarf.get_open_commitments`

```python
@mcp.tool(description=(
    "Get pending commitments — what has been promised, by whom, and by when. "
    "Only returns commitments with an explicit deadline (valid_until). "
    "Use for morning briefings, deadline tracking, and accountability queries."
))
def get_open_commitments(
    entity_name: str = None,
    before_date: str = None,
    format: str = "chronological",
) -> dict:
```

Parameters:
- `entity_name` — optional surface form. If provided, scoped to that entity. If omitted, returns all open commitments in the graph.
- `before_date` — optional ISO date. Only return commitments where `valid_until <= before_date`.
- `format` — `"chronological"` (default) or `"clustered"`

If `entity_name` is provided: resolve via `find_entity`. If not found, return `{"error": "not_found", "name": entity_name}`.

Calls `context_query.get_facts(entity_id, edge_label="ASSERTED", fact_type="commitment", include_stale=False)`. Post-filters to `valid_until` not null, and `valid_until <= before_date` if provided.

Returns:
```json
{
  "entity": {"id": "...", "name": "Marcus Webb", "type": "person"},
  "commitments": [<canonical fact objects, ASSERTED[commitment], valid_until in future>],
  "count": 3
}
```

If `entity_name` omitted, `entity` field is omitted and `commitments` spans all entities.

### 2. Register `pearscarf.get_open_blockers`

```python
@mcp.tool(description=(
    "Get what is currently blocked. "
    "Returns ASSERTED[blocker] facts that have no subsequent TRANSITIONED[resolution]. "
    "Use when preparing a status update or surfacing impediments."
))
def get_open_blockers(entity_name: str = None) -> dict:
```

Parameters:
- `entity_name` — optional. Scoped to entity if provided.

If `entity_name` is provided: resolve via `find_entity`. If not found, return `{"error": "not_found", "name": entity_name}`.

Calls `context_query.get_facts(entity_id, edge_label="ASSERTED", fact_type="blocker", include_stale=False)`. Post-filters in Python to exclude blockers that have a subsequent TRANSITIONED[resolution] edge from the same `from_entity` after the blocker's `source_at`.

Returns flat structure — no format parameter:
```json
{
  "entity": {"id": "...", "name": "Meridian Deal", "type": "project"},
  "blockers": [<canonical fact objects>],
  "count": 1
}
```

### 3. Register `pearscarf.get_recent_activity`

```python
@mcp.tool(description=(
    "Get what happened around an entity recently — transitions, references, and communications. "
    "Combines graph facts (state changes, mentions) with email metadata (who communicated with whom). "
    "Use when catching up on a deal, project, or person after time away."
))
def get_recent_activity(
    entity_name: str,
    since: str = None,
    format: str = "chronological",
) -> dict:
```

Parameters:
- `entity_name` — surface form (required). Resolved internally.
- `since` — ISO datetime string. Default: 7 days ago. Only return activity after this timestamp.
- `format` — `"chronological"` (default) or `"clustered"`

Resolves `entity_name`. If not found, returns `{"error": "not_found", "name": entity_name}`.

Calls three sources and merges:
1. `context_query.get_facts(entity_id, edge_label="TRANSITIONED", since=since)` — state changes
2. `context_query.get_facts(entity_id, edge_label="ASSERTED", fact_type="reference", since=since)` — mentions
3. `context_query.get_communications(entity_id, since=since)` — email metadata from Postgres

Merges results and sorts by `source_at` descending.

Returns:
```json
{
  "entity": {"id": "...", "name": "Marcus Webb", "type": "person"},
  "activity": [
    {
      "type": "fact",
      "edge_label": "TRANSITIONED",
      "fact_type": "status_change",
      "fact": "...",
      "source_at": "2026-03-28T10:00:00Z",
      "source_record": "change_007"
    },
    {
      "type": "communication",
      "sender": "marcus@acme.com",
      "recipient": "james@meridian.com",
      "subject": "Contract markup",
      "received_at": "2026-03-27T09:22:00Z",
      "record_id": "email_042"
    }
  ],
  "count": 8,
  "since": "2026-03-25T00:00:00Z"
}
```

The `type` field on each activity item distinguishes graph facts from communications — agents can filter or render them differently.

- Do not return commitments where `valid_until=null` — no deadline means no expiry signal, excluded by design
- Do not implement `get_entity_context` or `get_current_state` logic here — those are 1.15.4
