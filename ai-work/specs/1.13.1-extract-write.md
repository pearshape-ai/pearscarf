# Extraction prompt and indexer write path rewired to new fact model

## Context

- `graph.py` now accepts `edge_label`, `fact_type`, `source_at`, `valid_until` — but nothing is feeding it the new shape yet
- `extraction.md` and `ingest_extraction.md` still output a flat `category` field and `valid_at`
- `indexer._process_record` reads `fact.get("category")`, derives the Day node from `fact.get("valid_at")`, and only looks up `to_entity` by name in the current record's `entity_id_map` — if the entity wasn't extracted from this record, the fact is silently skipped
- This spec closes three gaps in one pass: prompt shape, indexer read path, and `to_entity` resolution

## Degradation principle

Facts are never dropped due to entity uncertainty on the `to` side. When a `to_entity` cannot be resolved — whether the name is missing, no candidates are found, the LLM judge returns ambiguous, or anything else unexpected — the fact degrades gracefully to Day node anchoring. The fact still has value. Only the `from_entity` blocks a record (via the existing `resolution_pending` path) because without a `from` node there is nothing to attach the fact to.

- `fact_type` values: `status_change`, `stage_change`, `role_change`, `ownership_change`, `resolution`, `completion`, `cancellation`, `other`

### 4. Update `indexer._process_record` — derive `source_at` from record timestamp

`source_at` is the event time. Derive it once per record before the fact loop from the record's own timestamp:

| Record type | `source_at` field |
|---|---|
| `email` | `emails.received_at` |
| `issue` | `issues.linear_created_at` |
| `issue_change` | `issue_changes.changed_at` |
| `ingest` | `records.created_at` |

If the type-specific field is empty, fall back to `records.created_at`. If that is also empty or null, use `_now()` — the indexing time becomes the event time. This is the `source_at == recorded_at` case: we don't know when it happened, so we treat it as happening when we indexed it. Log a warning when this fallback is used.

Pass `source_at` to every `create_fact_edge` call for this record.

### 5. Update `indexer._process_record` — `to_entity` resolution

Apply the degradation principle: every `to_entity` resolution path ends in either a node ID or a Day node — never a skip.

Resolution order:

```
1. Check entity_id_map — fast path, already resolved from this record's entities
2. If not found: call find_entity_candidates(entity_type, to_name, metadata)
3. Candidates found: call _resolve_entity_with_llm (same path as from_entity)
   - match    → use matched entity id; add to entity_id_map for reuse
   - new      → call create_entity; add to entity_id_map
   - ambiguous → log warning; fall through to Day node
4. No candidates: fall through to Day node
5. Any exception or unexpected result: log warning; fall through to Day node
```

`to_entity` type is not always known from the fact alone. Infer it from the entity name or default to searching across all types via `search_entities` with `entity_type=None`.

Add resolved `to_entity` ids to `entity_id_map` so they are reused if the same name appears in another fact in the same record.
