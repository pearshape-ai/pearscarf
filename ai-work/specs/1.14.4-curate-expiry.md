# Curator: expired commitment and promise detection

## Context

- ASSERTED[commitment] and ASSERTED[promise] facts carry a `valid_until` date when an explicit deadline was stated in the source record
- Once `valid_until` passes with no subsequent resolution, the commitment is either missed or quietly completed — the graph has no way to tell without a TRANSITIONED edge
- Expired commitments clutter the graph and produce false positives in `get_open_commitments` queries
- Unlike 1.14.2 and 1.14.3, this pass is not record-scoped — a commitment written months ago from a different record may expire today regardless of what new records are being indexed
- This check is purely mechanical — no LLM needed

## TODO

### 1. Add a global expiry scan to `curator.py`

Run once per curator cycle, independent of the record being processed. Place it at the end of the processing function, after the AFFILIATED and ASSERTED dedup passes.

The scan:

```
1. Query graph for all ASSERTED edges where:
   - fact_type IN ('commitment', 'promise')
   - valid_until IS NOT NULL
   - valid_until < today (local date, deployment timezone)
   - stale = false
2. For each expired edge:
   a. Call _notify_expiry(edge) — reserved hook, no-op for now
   b. Call mark_fact_stale(edge_id, replaced_by=null)
   c. Log the expiry
```

`today` is derived from the deployment timezone (same as Day node logic — `graph.utc_to_local_date(now())`).

### 2. Reserved notification hook `_notify_expiry(edge)`

Add a function `_notify_expiry(edge)` to `curator.py`. Body is a no-op for now — logs `"curator: expiry notification reserved for {edge_id}"` and returns.

This is the designated extension point for future HIL signaling, agent callbacks, or human-facing alerts when a commitment expires without resolution. The hook exists so the call site is already in place when that capability is added.

### 3. `mark_fact_stale` with no `replaced_by`

Expired commitments are staled with `replaced_by=null` — there is no successor edge. This is a legitimate state: the fact is stale not because it was superseded by a newer fact, but because its deadline passed.

Verify that `mark_fact_stale` in `graph.py` accepts `replaced_by=null` and writes it correctly. If the current signature requires a non-null `replaced_by`, update the signature to allow null.

### 4. Logging

One log entry per expired edge:

- `"curator: expired {fact_type} staled — edge_id={edge_id}, from={from_name}, valid_until={valid_until}, source_record={source_record}"`

## What NOT to do

- Do not check for a subsequent TRANSITIONED[completion] or TRANSITIONED[cancellation] before staling — stale on expiry unconditionally. Resolution detection is an augmentation agent concern.
- Do not implement the notification hook — reserve it, no-op only
- Do not scope this pass to the current record — it is a global scan across all records
- Do not run this pass before the dedup passes — expiry runs last in the curator cycle
- Do not touch ASSERTED facts with `valid_until=null` — no deadline, no expiry
- Do not touch TRANSITIONED edges — expiry only applies to ASSERTED[commitment] and ASSERTED[promise]
