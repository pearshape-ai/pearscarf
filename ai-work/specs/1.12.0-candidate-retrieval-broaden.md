# Entity resolution — candidate retrieval broadening

## Context

- Current `_resolve_entity()` in `indexer.py` only matches on exact name + email (person) or exact name + domain (company)
- This means "David", "Jim", "J. Whitfield", "Priya" all fail to find existing nodes and create duplicates
- `graph.find_entity()` and `graph.search_entities()` exist but are not used for fuzzy candidate retrieval
- This spec broadens retrieval so the resolution loop has the right candidates to judge against
- No LLM judge yet — this spec is purely about surfacing candidates from the graph

## TODO

### `graph.py` — new function `find_entity_candidates()`

- New function that returns a list of candidate entities (not just one) for a given name and context signals
- Searches in order of confidence, combining results without duplicates:
  - Exact name match (existing `find_entity()` behavior)
  - Email match for persons — if email is present in the extracted entity metadata
  - Domain match for companies — if domain is present in the extracted entity metadata
  - First name only match for persons — if the extracted name is a single token, search all Person nodes where `name` starts with that token
  - Substring name match — `search_entities()` with the extracted name as query
  - `IDENTIFIED_AS` edge match — find any entity that has an `IDENTIFIED_AS` edge where the surface form matches the extracted name (case-insensitive)
- Returns a list of candidate dicts, each with `id`, `type`, `name`, `metadata` — empty list if none found
- Deduplicates by entity ID — same node appearing from multiple search paths appears once

### `graph.py` — new Cypher query for `IDENTIFIED_AS` lookup

- Query: find Person or Company nodes that have an `IDENTIFIED_AS` relationship where `r.surface_form` matches the given name (case-insensitive)
- Returns entity node + element ID
- Used as the last step in `find_entity_candidates()`

### `indexer.py` — update `_resolve_entity()`

- Replace current exact-match-only logic with a call to `graph.find_entity_candidates()`
- If candidates list is empty → create new entity (same behavior as today)
- If candidates list has exactly one result and it is an exact name + email/domain match → use existing entity ID (same behavior as today, no regression)
- If candidates list has one or more results that are not exact matches → return the candidate list for the resolution loop to handle (1.12.3 will consume this)
- For now, if the resolution loop is not yet wired (pre-1.12.3), fall back to creating a new entity for non-exact matches — this is a temporary fallback, not the final behavior
- Log the candidate list at debug level for observability
