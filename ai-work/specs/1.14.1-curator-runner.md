## TODO

### 1. Create `curator.py`

A standalone script that owns the full worker loop and all curation logic. Mirrors the structure of `indexer.py`.

The loop:

```
while True:
    1. Reset timed-out claims (crash recovery)
    2. Claim one entry from curator_queue (oldest queued_at first)
       None available → sleep poll_interval, continue
    3. Process the claimed record_id inline
    4. Delete the queue entry
    5. Sleep poll_interval, repeat
```

No threads. No spawning. One entry at a time, inline.

### 2. Claiming an entry

Select one row where `claimed_at IS NULL`, ordered by `queued_at ASC`. Set `claimed_at = now()` in the same update. Returns the `record_id`, or `None` if the queue is empty.

### 3. Timeout recovery

Before each claim attempt, find rows where `claimed_at IS NOT NULL` and `claimed_at < now() - claim_timeout`. Reset `claimed_at = null` on each. Log with `record_id` and original `claimed_at`.

Default `claim_timeout`: 10 minutes. Handles the case where `curator.py` crashes mid-processing — the entry is automatically retried on next startup.

### 4. Processing (stub for now)

The processing function takes a `record_id`, logs `"curator: processing {record_id}"`, and returns. Subsequent specs (1.14.2–1.14.5) fill in the actual graph logic here.

On exception:
- Handled exception → log warning with `record_id`, delete queue entry (don't retry logic errors)
- Unhandled exception → log error, release claim (`claimed_at = null`), let timeout recovery handle retry

### 5. Configuration

Configurable via environment variables, following the same convention as `indexer.py`:

| Variable | Default | Meaning |
|---|---|---|
| `CURATOR_POLL_INTERVAL` | `30` | Seconds between poll cycles |
| `CURATOR_CLAIM_TIMEOUT` | `600` | Seconds before an abandoned claim is reset |
