# Context

- Gmail expert is the first data source, fully working with observer mode (polling) and interactive mode (worker delegation).
- Linear is the second data source. It validates the architecture works across heterogeneous data.
- Linear has a GraphQL API. We have the API key. No MCP — direct API calls.
- This iteration builds the expert with full read/write capabilities and a polling loop. No graph writes, no extraction. Just the expert agent that can operate Linear and a polling loop that saves issues to the System of Record.

## Linear expert agent

An expert agent following the same pattern as the Gmail expert. The worker delegates Linear tasks to it, it executes via the GraphQL API, and replies with results.

Interactive tools — the expert should be able to:
- List issues (filter by status, assignee, project, priority, label)
- Get a specific issue by identifier (e.g. "ENG-42") with full details
- Create an issue (title, description, assignee, priority, project, labels)
- Update an issue (change status, priority, assignee, labels)
- Add a comment to an issue
- Search issues by text query

Each tool calls Linear's GraphQL API directly. The expert has a system prompt (in `pearscaff/prompts/linear-expert.md`) describing its role and available tools.

## System of Record — issues table

Add an `issues` table in Postgres following the same pattern as `emails`:

- `record_id` referencing the base `records` table
- `linear_id` — Linear's unique ID (for dedup and sync)
- `identifier` — human-readable ID like "ENG-42"
- `title`, `status`, `priority`, `assignee`, `project`, `labels`, `url`
- `created_at`, `updated_at` from Linear

`store.py` gets `save_issue`, `get_issue`, `get_issue_by_linear_id`, `list_issues` functions.

Dedup on `linear_id`, same as emails dedup on `message_id`.

## Polling loop

Same pattern as Gmail polling. A background thread that syncs issues from Linear on an interval.

First run: load all active issues (and recently completed). Each becomes a record in the System of Record. Store a `synced_at` cursor.

Subsequent runs: query Linear for issues updated since `synced_at`. Update existing records, create new ones. Advance the cursor.

New/updated issues get saved to the SOR and sent to the worker for triage, same flow as email polling — the worker decides if it's relevant and asks the human when uncertain.

Polling is opt-in via a CLI flag, same as `--poll-email`.

## Config

- `LINEAR_API_KEY` — the API key
- `LINEAR_POLL_INTERVAL` — seconds between polls, default 300
- `LINEAR_TEAM_ID` — optional, to scope polling to a specific team
