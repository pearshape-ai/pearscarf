# Ingest expert — full implementation

## Context

- v1.11.0 scaffolded the ingest expert with empty tool stubs and no storage wiring
- This version fully implements the ingest expert tools and wires them into the pipeline
- Two record paths exist: seed files (type `ingest`) and typed JSON record files (email, issue, issue_change)
- The indexer is the only writer to the graph — ingest records flow through the same pipeline as all other records
- The indexer needs a dedicated branch for `ingest` records, using a separate extraction prompt instead of the standard one
- Ingest records bypass triage — they are auto-classified as `relevant` on save, same as `issue_change` records today

## TODO

### Storage — ingest record type

- `store.py` gets a new `save_ingest()` function that writes a record of type `ingest` to the `records` table with `classification = 'relevant'`
- Raw seed file content stored in the `raw` field of the `records` table — no separate typed table needed
- `ingest` is a valid `type` value in the `records` table — no schema change required, the column is free text

### Ingest expert tools — full implementation

- `parse_seed` tool: reads the `.md` file at the given path, calls `store.save_ingest()` with the raw content, returns the assigned `record_id`
- `parse_record_file` tool: reads the JSON file at the given path, expects either a single record dict or a list of record dicts, routes to the correct `store` function based on `record_type` — `save_email()` for `email`, `save_issue()` for `issue`, `save_issue_change()` for `issue_change` — returns count of records saved and any duplicate skips
- Both tools now have fully implemented `execute()` bodies
- `parse_record_file` sets `classification = 'relevant'` on all saved records so they are picked up by the indexer without triage

### Indexer — ingest branch

- `_build_content()` in `indexer.py` gets an `ingest` branch — reads the `raw` field from the `records` table directly, returns it as the content string
- `_extract()` in `indexer.py` gets type-aware prompt selection — when `record_type == 'ingest'`, loads `prompts/ingest_extraction.md` instead of `prompts/extraction.md`
- All other record types continue using `prompts/extraction.md` unchanged

### New prompt — `pearscarf/prompts/ingest_extraction.md`

- Dedicated extraction prompt for seed ingest records
- Input is a typed block format seed file with `## people`, `## companies`, `## projects`, `## facts` sections
- Output is the same JSON shape as the standard extraction prompt: `entities` array and `facts` array
- Instructs the LLM to treat all content as declared ground truth — `confidence: stated` for everything, no inference
- People entries produce `person` entities with `email` and `role` metadata, plus `WORKS_AT` facts from the `## facts` section
- Company entries produce `company` entities with `domain` metadata
- Project entries produce `project` entities
- Facts entries produce fact edges with the declared category
