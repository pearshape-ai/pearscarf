# Entity resolution — resolution loop wired into indexer

## Context

- 1.12.0 produces candidate lists, 1.12.1 builds context packages, 1.12.2 implements the LLM judge
- This spec wires all three into `_process_record()` replacing the current `_resolve_entity()` stub and its temporary fallback
- After this spec, the indexer makes real resolution decisions for every extracted entity before writing facts to the graph

## TODO

### `indexer.py` — new `_resolve_entity()` implementation

Replace the current method entirely. New logic in order:

- Call `graph.find_entity_candidates()` with the extracted entity name, type, and metadata
- If no candidates → create new entity via `graph.create_entity()`, return new ID
- If candidates contain an exact name + email/domain match → use that entity ID directly, skip LLM call
- Otherwise → call `graph.get_entity_context()` for each candidate to build context packages, then call `_resolve_entity_with_llm()` with the extracted entity, source record context, and candidate context packages
- On `match` → return the matched candidate ID
- On `new` → create new entity, return new ID
- On `ambiguous` → flag the entity as unresolved, store the ambiguity state (extracted entity, candidate IDs, reasoning) on the record, return `None`

### `indexer.py` — ambiguity state on records

- When resolution returns `ambiguous`, store the ambiguity state in a new Postgres column `resolution_pending` (JSON) on the `records` table
- `resolution_pending` stores a list of unresolved entities for that record, each with: surface form, type, candidate IDs, reasoning
- Record continues processing — facts that reference an unresolved entity (where entity ID is `None`) are skipped for now
- Record is not marked `indexed = TRUE` until all entities are resolved — it stays in a pending state

### `db.py` — schema addition

- Add `resolution_pending JSONB` column to `records` table via `ALTER TABLE IF NOT EXISTS` — safe to run on existing data, defaults to `NULL`
- Add `resolution_status TEXT` column to `records` table — values: `NULL` (not started), `pending` (has unresolved entities), `complete` (all resolved)

### `indexer.py` — source record context string

- Build a short context string from the current record for the LLM judge — sender, recipients, subject, first 300 chars of body for emails; identifier, title, assignee, project for issues; field, from/to values for issue changes
- Reuses fields already fetched in `_build_content()` — no extra DB queries
