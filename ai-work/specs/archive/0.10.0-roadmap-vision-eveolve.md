# Context

- Core agent system works: worker, experts, sessions, triage, indexer, retriever.
- Storage quality needs improvement — facts table output was poor, entity resolution is fragile.
- Deciding to focus on getting the storage and context layer right before adding more agents.
- Transport layer for experts (headless browser vs MCP vs API) is a separate concern — experts are black boxes to the rest of the system. Picking the right transport per expert is a pragmatic choice, not an ideological one.

## Update docs/vision.md

Tone down the anti-MCP stance. Replace the thesis framing with something more balanced:

- The core architectural principle is separation of concerns: experts own their domains, the worker reasons, the storage layer connects everything.
- How an expert accesses its domain (headless browser, MCP, raw API, whatever) is an implementation detail. The system doesn't care. Pick the right tool for the job.
- The real differentiator is the context layer — how well heterogeneous data gets connected, stored, and surfaced to models when they need it.
- MCPs, APIs, headless browsers are all valid transport mechanisms. The architecture supports swapping between them without touching anything outside the expert.
