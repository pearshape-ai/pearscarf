# Context

- ChromaDB integration is in place (v0.7.0). Indexer can embed records.
- This version adds email triage — the worker classifies incoming emails as relevant or noise.
- Auto-classifies when confident, asks the human when uncertain.
- All classification activity is visible on both Discord and REPL.
- Human responses during classification teach the system and build context.
- Indexer only processes records classified as relevant.

## Worker Triage Flow

When the worker receives a new email from the Gmail expert:

```
Step 1: Check sender against known entities in the graph
    ↓
If sender is known entity → auto-classify as relevant
    ↓
Step 2: Check for obvious noise signals (no-reply address, unsubscribe link, promotional keywords)
    ↓
If obvious noise → auto-classify as noise
    ↓
Step 3: If uncertain → ask human
    ↓
Human responds → worker stores classification + human context
```

## Human Context Capture

When the human responds to a classification question, the worker:

1. Stores the classification on the record (`relevant` or `noise`)
2. Stores the human's reasoning in `classification_reason`
3. Stores any additional context in `human_context` — this gets passed to the Indexer alongside the email content so entity extraction is richer
4. Prints confirmation of what it stored

## Override

If the human disagrees with an auto-classification, they can respond in the same session/thread:

```
worker > ✗ Noise: Email from deals@vendorx.com "Renewal notice"

you > no that's relevant, VendorX is our hosting provider

worker > ✓ Reclassified as relevant. Stored: VendorX → hosting provider.
```

Worker updates the record's classification and stores the human context.

## Failure Visibility

If any step fails, it's printed clearly:

```
worker > ⚠ Failed to classify email_71: LLM call timed out. Record left as pending.
worker > ⚠ Failed to store classification for email_72: database write error.
```

Failed records stay as `classification = 'pending'` and can be retried.
