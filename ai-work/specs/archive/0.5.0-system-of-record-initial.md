# Context

- Worker-expert communication is working. Gmail expert reads emails via headless browser.
- This version adds the System of Record — persistent structured storage for all data flowing through the system.
- Each expert agent owns writing to its domain-specific tables. The worker reads from storage but never writes raw records.

## Database Schema

```sql
-- Base record table, shared across all data types
records (
    id TEXT PRIMARY KEY,
    type TEXT NOT NULL,           -- "email", "issue", "event", etc.
    source TEXT NOT NULL,         -- "gmail_expert", "linear_expert", etc.
    created_at TEXT NOT NULL,
    raw TEXT                      -- original unprocessed content from the expert
)

-- Email-specific table, owned by gmail_expert
emails (
    record_id TEXT PRIMARY KEY REFERENCES records(id),
    message_id TEXT UNIQUE,       -- Gmail's unique message ID, used for deduplication
    sender TEXT,
    recipient TEXT,
    subject TEXT,
    body TEXT,
    received_at TEXT
)
```

## Data Flow

```
Gmail expert reads email via headless browser
    ↓
Gmail expert writes to records + emails table
    ↓
Gmail expert sends full message to worker:
    { action: "new_email", record_id: "email_71", sender: "...", subject: "...", body: "...", summary: "..." }
    ↓
Worker receives full data + record_id, reasons and responds to human
```

## Ownership

- **Expert agents write** to the records table and their own typed table. Gmail expert writes emails. Future Linear expert writes issues.
- **Worker agent reads** from the System of Record when it needs context. Worker never writes raw records.
- Each expert is the authority for its domain data. It has the raw content, it knows the structure, it handles storage.

## Deduplication

- Gmail expert checks `emails.message_id` before inserting. If the email has already been stored, skip the insert and don't send a duplicate message to the worker.
- Each future expert implements its own deduplication logic using the appropriate unique identifier for its domain.
