# Context

- Day nodes exist (v1.8.2). Entity nodes exist (Person, Company, Project, Event).
- The old graph model stored facts as separate constructs (Fact nodes or properties). The new model: everything is an edge. Entities and Days are the only node types. Facts are labeled edges between them.
- A fact-edge connects either two entity nodes or one entity node to a Day node. The edge label is the fact category. The edge carries temporal metadata and a reference to the source record.
- This version defines the edge schema (what properties every fact-edge carries) and the fact categories (what labels are available).

## Define fact-edge properties

Every fact-edge in Neo4j carries these properties:

- `fact` — the human-readable fact text (e.g. "contract is in legal review", "presented the Q3 demo")
- `category` — the fact category label (see below)
- `confidence` — stated or inferred
- `source_record` — the record ID that produced this fact
- `source_type` — email, issue, issue_change, etc.
- `valid_at` — UTC timestamp, when this fact became true in the real world (from the record's date, or explicitly stated in the content)
- `created_at` — UTC timestamp, when PearScaff recorded this fact
- `invalid_at` — UTC timestamp, when this fact stopped being true (null if still current)

## Define fact categories

These are the Neo4j relationship types used for fact-edges. They replace the old separate "relationship" and "fact" distinction — everything is a fact-edge now, some are structural (like works_at) and some are temporal claims (like committed_to).

**Structural (stable, rarely invalidated):**
- `WORKS_AT` — person → company
- `FOUNDED` — person → company
- `MANAGES` — person → project
- `PART_OF` — project → company, or company → company (subsidiary)
- `MEMBER_OF` — person → project/event

**Activity (time-bound, connects to Day node or between entities):**
- `COMMUNICATED` — person → person, or person → company (email exchange, meeting)
- `MENTIONED_IN` — entity → Day (entity appeared in a record on this day)
- `STATUS_CHANGED` — project/entity → Day (status update, stage transition)

**Claims (business facts with temporal significance):**
- `COMMITTED_TO` — entity → entity or entity → Day (deadline, deliverable, promise)
- `DECIDED` — entity → Day (decision made)
- `BLOCKED_BY` — entity → entity (dependency, blocker)
- `EVALUATED` — entity → entity (considering, reviewing, comparing)

**Meta:**
- `IDENTIFIED_AS` — entity → entity (alias resolution, "Mike Chen" identified as "Michael Chen")

The extraction prompt will output the category. If a fact doesn't fit neatly into a category, use `MENTIONED_IN` as the fallback — it's the most general.

## Add `create_fact_edge` to `graph.py`

- Takes: from_node_id, to_node_id, category, fact_text, confidence, source_record, source_type, valid_at
- Creates a Neo4j relationship of the given category type between the two nodes
- Sets all properties on the edge
- `created_at` is set automatically to now (UTC)
- `invalid_at` is set to null
- Returns the edge

## Add `invalidate_fact_edge` to `graph.py`

- Takes: edge_id or matching criteria (from_node, to_node, category, fact_text)
- Sets `invalid_at` to now (UTC) on the matching edge
- Does not delete the edge — history is preserved
- This is used when a newer fact supersedes an older one

## Add `get_facts_for_entity` to `graph.py`

- Takes: entity name or node ID, optional `include_invalid` flag (default false)
- Returns all fact-edges connected to this entity
- By default filters to current facts only (where `invalid_at` is null)
- When `include_invalid=True`, returns full history

## Add `get_facts_for_day` to `graph.py`

- Takes: date string (ISO format)
- Returns all fact-edges connected to that Day node
- Note: this only captures single-entity facts anchored to the Day node, not two-entity facts that happened on that day (those are found via `valid_at` filtering)
