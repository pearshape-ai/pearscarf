## Wire graph.py to Neo4j

Replace the stubbed function bodies in `graph.py` with Neo4j reads and writes via the `neo4j` Python driver. The module becomes the interface between the application and the Neo4j graph.

Entity types map to node labels in Neo4j. A person entity becomes a node with label `Person`, a company becomes `Company`, etc. Metadata fields become properties on the node. Every node gets a `name` property and a `created_at` timestamp.

Relationships become Neo4j relationships between nodes. The `type` from extraction (works_at, customer_of, etc.) becomes the relationship type. Every relationship gets a `source_record` property pointing back to the email it came from and a `created_at` timestamp.

Facts become Neo4j relationships as well — an edge from the entity to a `Fact` node, or a property on a relationship edge. The decision on fact storage shape can be made during planning, but the key requirement: facts must carry the claim text, confidence level, source record, and created_at timestamp.

Entity resolution: before creating a new node, check if one already exists with the same name and type. For persons, also match on email. For companies, also match on domain. If found, reuse the existing node and add any new metadata. If not found, create a new node.

`list_entity_types` stays as-is — it reads from Postgres entity_types table, which is still useful for building the extraction prompt.
