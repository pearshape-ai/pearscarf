# Context

- The terminal REPL needs UX improvements: non-blocking input, clear attribution of who is speaking, and activity indicators.

# Problems

1. **Prompt blocks while agents are working.** After sending a message, the REPL hangs until the full response comes back. Human can't switch sessions or send another message.
2. **No attribution on messages.** Unclear who is typing and which agent is doing work underneath.
3. **No activity indicator.** No way to tell if the system is thinking, waiting on an expert, or stuck.

# TODO

## Non-blocking input

- The REPL prompt should always be available for input, regardless of whether agents are processing.
- Agent responses stream in above the prompt line as they arrive.
- Human can type and send new messages at any time, even mid-response.

## Message attribution

Every message displayed in the REPL should show who sent it and which session it belongs to:

```
[ses_006] you > read my latest unread email, post it here and mark as read

[ses_006] worker > thinking...
[ses_006] worker > forwarding to gmail_expert
[ses_006] gmail_expert > reading latest unread email...
[ses_006] gmail_expert > marking as read...
[ses_006] worker > 

  **Latest Unread Email (Now Marked as Read)**
  From: Venture Forum <john@ventureforum.net>
  Subject: FAQ: Your Last Day to Save $150
  ...

[ses_006] you > 
```

## Activity indicator

- When an agent is working, show a live status line above the prompt:
  ```
  [ses_006] gmail_expert working... (3s)
  [ses_006] you > 
  ```
- The status line updates as work moves between agents:
  ```
  [ses_006] worker thinking... (1s)
  [ses_006] worker → gmail_expert... (2s)
  [ses_006] gmail_expert working... (5s)
  [ses_006] worker composing response... (1s)
  ```
- Status line disappears once the response is delivered.

## What NOT to build yet

- No rich TUI framework (curses, blessed, ink) — keep it simple with terminal escape codes if needed.
- No parallel message rendering — messages appear in order, one at a time.
