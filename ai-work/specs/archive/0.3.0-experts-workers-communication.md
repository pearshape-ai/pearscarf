# Context
Agent Communication System — Implementation Spec:
- The expert and worker agents are standalone, an agent-to-agent communication system needs to be implemented
- Async agent-to-agent communication system with sessions. Two agent types: 
    - worker (context-savvy, reasoning, task routing) and 
    - expert agents (UI-operating specialists, starting with Gmail). 
- SQLite is both the message transport and the system of record. 
- No external queue infrastructure.

## Architecture

```
Terminal REPL (human interface)
    ↓
Worker Agent (context, reasoning, task routing)
    ↓ async messages via SQLite
Expert Agents (headless browser UI operators, starting with Gmail)
```

## Message Flow

### Human-initiated (request/response)
1. Human types command in REPL → message inserted as `from: "human", to: "worker"`
2. Worker picks it up, reasons about it, may forward to expert → message `from: "worker", to: "gmail_expert"`
3. Expert processes, responds → message `from: "gmail_expert", to: "worker"`
4. Worker processes result, responds to human → message `from: "worker", to: "human"`
5. REPL displays worker's response

### Expert-initiated (event)
1. Expert detects something (e.g. new urgent email) → creates new session, sends message `from: "gmail_expert", to: "worker"`
2. Worker picks it up, decides if human needs to know → message `from: "worker", to: "human"`
3. REPL prints notification with new session ID
4. Human can switch to that session to interact further

## Agent Polling

Agents poll SQLite for unread messages addressed to them:

After processing a message, mark it as read and write response as a new row. Polling interval: 1 second.

## Discord Bot

- Each session maps 1:1 to a Discord thread.
- A new message in the main channel creates a new session and a new Discord thread for it.
- All follow-up messages within that session happen inside the corresponding thread.
- When an expert creates a session (e.g. incoming email event), a new thread is automatically created in a designated channel with the session summary as the thread name.
- Human replies inside a thread are automatically tagged with that thread's session ID — no need to specify it manually.
- Sessions are never closed. They persist and can be resumed at any time by posting in the thread.

## Terminal REPL

### Prompt format
```
[ses_001] > 
```

Active session ID shown in prompt. All input gets stamped with the current session ID.

### Commands
```
/sessions        — list all active sessions (id, initiated_by, summary)
/switch <id>     — switch active session
/new             — create a new session manually
/history         — print all messages in current session, in order
/history <id>    — print all messages in a specific session
```

### Incoming event notification
When an expert creates a new session, print an interrupt in the terminal:
```
--- NEW SESSION ses_003: gmail_expert — urgent email from investor ---
```
Human stays in current session unless they `/switch`.

## Message Data Format

The `data` field is flexible JSON. Examples by action:

```json
// action: "read_unread"
{ "limit": 5 }

// action: "read_unread" response
{ "emails": [{ "id": "...", "from": "...", "subject": "...", "snippet": "...", "timestamp": "..." }] }

// action: "mark_as_read"
{ "email_ids": ["id1", "id2"] }

// action: "new_email" (expert-initiated event)
{ "email": { "id": "...", "from": "...", "subject": "...", "snippet": "...", "urgent": true } }
```

## Reasoning Field

Every message should include a `reasoning` field explaining why the agent sent it. This is for debugging, evaluation, and hallucination detection. Examples:
- "User requested last 5 unread emails, forwarding to gmail_expert"
- "Email from investor@acme.com matches VIP list, flagging as urgent"
- "Marking as read per human request in ses_001"

## What NOT to build yet
- No real queue infrastructure (Redis, RabbitMQ, etc.)
- No websockets or push — polling only
- No authentication between agents
- No retry/dead-letter logic — just log failures
- No multi-worker support — single worker only
- No context engine or summarization — just raw message passing for now
