## Fact writing logic (step 5)

For each fact in the extraction output:

**Two-entity fact** (`to_entity` is not null):
- Resolve `from_entity` node via `find_entity` or `create_entity` (should already exist from step 4)
- Resolve `to_entity` node the same way
- Call `create_fact_edge(from_node, to_node, category, fact_text, confidence, source_record, source_type, valid_at)`
- `source_record` = the record ID being processed
- `source_type` = the record type (email, issue, issue_change)
- `valid_at` = use the fact's `valid_at` if present, otherwise use the record's timestamp

**Single-entity fact** (`to_entity` is null):
- Resolve `from_entity` node
- Determine the Day node date: if the fact has `valid_at`, use that date; otherwise convert the record's UTC timestamp to local date via `utc_to_local_date`
- Call `get_or_create_day(date_string)` to get the Day node
- Call `create_fact_edge(from_node, day_node, category, fact_text, confidence, source_record, source_type, valid_at)`

## Handle entity name mismatches gracefully

The extraction output may reference a `from_entity` or `to_entity` name that doesn't exactly match any entity in the `entities` array (the test script already flags this). When a fact references an entity name that wasn't in the entities array:
- Log a warning with the mismatched name and the record ID
- Skip that fact — don't crash, don't create orphan edges
- This is an extraction quality signal, not a runtime error
