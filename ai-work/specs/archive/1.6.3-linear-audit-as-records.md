# Context

- Linear issues flow through extraction into Neo4j and Qdrant (v1.6.2). Cross-source entity resolution is working.
- Currently polling captures current state snapshots. An issue that moved through multiple statuses only shows the final state.
- The graph needs to know that transitions happened — these are facts about how work progressed. But the temporal storage model (bi-temporal timestamps, invalidation) is a separate concern handled in v1.7.0.
- This version captures the audit data and feeds it through extraction as regular facts — same storage mechanism as everything else.

# TODO

## Capture issue change history from Linear

Linear's GraphQL API exposes history entries for each issue — what field changed, old value, new value, who changed it, when. The polling loop should fetch these changes alongside the current state sync.

Each meaningful change (status change, assignee change, priority change) becomes its own record in the SOR — a new record type (e.g. "issue_change") with a typed table. It records: which issue, what field changed, old value, new value, who changed it, when.

This is separate from the issue record itself. The issue record stays as the current snapshot. Change records are the timeline.

## Feed changes through extraction

Change records flow through the Indexer's extraction pipeline like any other record. A status change from "In Progress" to "In Review" produces a fact: "Acme API Integration moved to In Review" attributed to the project entity, with the person who made the change.

Change records are structured and short — not free-text like emails. The `_build_content` pattern needs an equivalent for change records, assembling the change into a text block the extraction prompt can work with.

If the extraction prompt needs change-specific guidance (e.g. "for change records, extract the fact about what changed, don't look for new entities"), add a small section conditioned on record_type.

## What stays the same

- Facts are stored as regular edges in Neo4j — same `upsert_fact`, same Fact nodes, same `HAS_FACT` relationships. No `valid_at` / `invalid_at` yet.
- Existing extraction pipeline, entity resolution, Qdrant embedding — all untouched.
- Linear expert interactive tools and current-state polling — untouched.
