# Context

- PearScaff makes LLM calls across multiple agents: worker, Gmail expert, Indexer, Retriever.
- Mem0 adds its own LLM calls internally for extraction and conflict resolution.
- Current observability is session.log — flat text, good for debugging individual interactions, no aggregate visibility.
- Need to see the full token and cost picture across all agents including Mem0's internal calls.

# TODO

## LangSmith Integration

LangSmith becomes the primary observability layer for the system. Every LLM call — ours and Mem0's — becomes a traced run with full prompt, response, token counts, latency, and cost.

### Tracing

Every agent turn is a traced run. Traces are hierarchical — a single email triage shows the worker's reasoning, the Retriever's search, and any Mem0 calls as child spans under one parent trace. This gives us the full picture of what happened, in what order, and what it cost.

For Mem0's internal calls, we need to pass a LangSmith-wrapped LLM client to Mem0's config so its extraction and conflict resolution calls appear in our traces. If that's not possible, we at minimum trace the Mem0 add and search calls as parent spans.

### Cost Tracking

LangSmith computes cost from token counts and model pricing automatically. We want cost broken down by agent (worker, gmail_expert, indexer, retriever), by operation type (triage, retrieval, extraction), and over time (daily, weekly). This tells us whether the system is economically viable and where to optimize.

### Alerts

Set thresholds on daily token spend or cost. If the polling loop processes 200 emails overnight or Mem0 makes excessive conflict resolution calls, we get notified before the bill surprises us.

### Tagging

Every trace is tagged with: agent name, session ID, record ID, operation type. This lets us filter by any dimension — "show me all Mem0 extraction calls for emails from Acme" or "what's the average cost of a triage decision."

### What We Expect to Learn

Once running with real email traffic:

- How much does it cost to triage one email end-to-end?
- How many LLM calls does Mem0 make per email on average?
- Which agent consumes the most tokens?
- Are there emails that are disproportionately expensive to process?
- What does the Retriever's context package actually look like for different query types?

### Relationship to session.log

session.log stays as a local fallback — offline debugging, grepping, or when LangSmith is unreachable. LangSmith is additive. The log format and content don't change.

### Configuration

LangSmith is opt-in. When tracing is not enabled, the system runs exactly as before.

## What NOT to build

- No cost enforcement (stopping agents when budget exceeded) — LangSmith reports and alerts, doesn't enforce
- No quality evaluation — LangSmith shows what the LLM produced, doesn't judge whether it was correct
- No Langfuse integration yet — if we want to move off SaaS later, the integration pattern is similar and switching is feasible
