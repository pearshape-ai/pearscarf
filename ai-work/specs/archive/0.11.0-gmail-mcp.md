
### Expert Tools via MCP

The Gmail expert gets MCP-backed versions of its core operations:

- List unread emails
- Read email by ID
- Search emails by query
- Mark as read

These supplement (not replace) the existing headless browser tools. The expert can use either transport depending on the task.

### Transport Priority

MCP is the default for all standard email operations (list, read, search, mark as read). If `GMAIL_CLIENT_ID` and related credentials are configured in `.env`, these operations go through MCP automatically. No LLM decision involved.

Headless browser is used when:
- MCP credentials are not configured (backward compatible — system works as before)
- MCP auth fails and can't be refreshed (graceful fallback)
- A task requires complex UI interaction that MCP tools don't cover

The expert checks for MCP availability at startup and logs which transport is active:

```
[gmail_expert] [--] [action] MCP transport active (OAuth configured)
```
or
```
[gmail_expert] [--] [action] Browser transport active (no MCP credentials)
```

## Email Polling Loop

Opt-in background loop that checks for new unread emails on an interval.

### Activation

Polling is off by default. Enable via CLI flag:

```bash
pearscaff run --poll-email
pearscaff discord --poll-email
```

Without the flag, the expert only reads emails when asked.

### Configuration

```
GMAIL_POLL_INTERVAL=300  # seconds, default 5 minutes
```

### How It Works

```
Every GMAIL_POLL_INTERVAL seconds:
    |
    Expert calls MCP: list unread emails
    |
    Compare message_ids against System of Record
    |
    For each new email:
        Store raw record + email (existing save_email flow)
        Create new session
        Send full data to worker
    |
    For known emails: skip
```

Deduplication uses the existing `message_id` UNIQUE constraint on the emails table. No new dedup logic needed.

### Notifications

New emails detected by polling trigger the same visibility as manually-read emails:

Discord:
```
[Thread: New email from michael@acmecorp.com]
worker > New email from michael@acmecorp.com
         Subject: "Q3 Invoice"
         Is this relevant and why?
```

REPL:
```
--- NEW SESSION ses_014: worker — new email from michael@acmecorp.com ---
[ses_014] worker > New email from michael@acmecorp.com
                   Subject: "Q3 Invoice"
                   Is this relevant and why?
```

### Multiple Emails

If 3 new emails arrive between polls, each gets its own session and notification. One session per email, same as the existing architecture.

### Error Handling

```
worker > ⚠ Email poll failed: OAuth token expired. Re-authenticate with: pearscaff gmail --auth
worker > ⚠ Email poll failed: MCP server unreachable. Retrying in 60s.
```

Errors printed on Discord/REPL. Poll continues on next interval. Failed polls don't crash the system.

## Gmail Auth Command

New CLI command for OAuth setup:

```bash
pearscaff gmail --auth
```

Runs the OAuth flow, stores refresh token. Replaces the browser-based login for MCP operations. The existing `pearscaff expert gmail --login` stays for headless browser sessions.
