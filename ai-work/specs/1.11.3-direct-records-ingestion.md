# Record ingestion — direct insert, strict schema

## Context

- v1.11.1 implemented `parse_record_file` as a stub
- Record ingestion does not need LLM involvement — JSON files contain structured data that maps directly to the store schema
- `parse_record_file` should be a direct insert tool: validate schema, insert, done
- The ingest expert accepts a folder (like the eval dataset `data/` structure) or a single file, plus an explicit `--type` flag
- If no `--type` is passed, the tool does not execute
- All records in a batch must pass schema validation before any inserts happen — one failure aborts the entire batch
- Input JSON must use exact store field names — no mapping, no inference, no LLM

## TODO

### `parse_record_file` tool — full implementation

- Accepts `file_path` (file or folder) and `record_type` (`email`, `issue`, `issue_change`)
- If `file_path` is a folder, loads all `.json` files in it recursively — same walk pattern as `eval_runner._load_records()`
- If `file_path` is a single file, loads that file only
- **Validation pass first**: iterate all records, validate each against the expected schema for the given `record_type` — collect all errors before inserting anything
- If any record fails validation, abort with a full error report — zero inserts
- If all records pass, insert each via the appropriate store function:
  - `email` → `store.save_email()`
  - `issue` → `store.save_issue()`
  - `issue_change` → `store.save_issue_change()`
- Returns: count inserted, count skipped as duplicates, list of record IDs assigned

### Expected schemas (exact store field names)

**email** — required fields: `sender`, `subject`, `body`. Optional: `recipient`, `message_id`, `received_at`.

**issue** — required fields: `linear_id`, `title`. Optional: `identifier`, `description`, `status`, `priority`, `assignee`, `project`, `labels`, `comments`, `url`, `linear_created_at`, `linear_updated_at`.

**issue_change** — required fields: `issue_record_id`, `field`, `changed_at`. Optional: `from_value`, `to_value`, `linear_history_id`, `changed_by`.
