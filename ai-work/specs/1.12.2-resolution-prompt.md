# Entity resolution — resolution prompt

## Context

- 1.12.0 and 1.12.1 produce a candidate list with rich context packages
- This spec defines the LLM judge: a dedicated prompt that takes an extracted entity + source record context + candidate context packages and decides: match, new, or ambiguous
- Follows the same pattern as `prompts/ingest_extraction.md` — focused task, structured JSON output, no preamble
- The prompt must handle the three-way decision cleanly and provide reasoning the system can log and the human can inspect

## TODO

### New prompt `pearscarf/prompts/entity_resolution.md`

Input sections the prompt receives:

- **Extracted entity** — name, type, metadata (email, role, domain if present)
- **Source record context** — the record content snippet (sender, recipients, subject, a short body excerpt) that produced this entity mention — gives the judge contextual signals like company domain, thread participants, project references
- **Candidate list** — each candidate includes: name, type, metadata, their current facts (category + fact text), their direct connections (entity name + category)

Output — structured JSON, one of three shapes:

```
{"decision": "match", "candidate_id": "<entity element ID>", "reasoning": "..."}
{"decision": "new", "reasoning": "..."}
{"decision": "ambiguous", "reasoning": "...", "candidate_ids": ["...", "..."]}
```

Prompt rules to encode:

- `match` — use when one candidate clearly refers to the same real-world entity as the extracted mention, based on name similarity plus at least one corroborating signal (shared email domain, shared company, shared project, overlapping connections)
- `new` — use when no candidate is a plausible match, or when candidates exist but the context clearly places this entity in a different company or project context
- `ambiguous` — use when two or more candidates are plausible and context signals do not disambiguate; also use when there is one candidate but the match evidence is weak and could go either way
- Always provide `reasoning` — one or two sentences explaining the decision
- Prefer `new` over `ambiguous` when candidates exist but the source record context clearly rules them out
- Prefer `match` over `ambiguous` when email or domain is an exact match — email is deterministic, no ambiguity possible
