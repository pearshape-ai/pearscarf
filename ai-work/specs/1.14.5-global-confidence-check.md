# Curator: global confidence upgrade pass and agent documentation

## Context

- 1.14.2 and 1.14.3 include a confidence upgrade step that runs after a dedup collapse — if any merged source record produced the fact as `stated`, the surviving edge is upgraded from `inferred` to `stated`
- But this upgrade only fires when a collapse happens. There is a gap: if a literal dup merge adds a `stated` source record to an existing edge (without triggering a collapse), the confidence upgrade never runs
- Concretely: `record_001` writes edge X as `inferred`. `record_007` indexes — literal dup detected, no new edge written, but `source_records` becomes `["record_001", "record_007"]` where `record_007` produced the fact as `stated`. The curator for `record_007` finds no duplicates to collapse and exits without upgrading the confidence.
- A global pass closes this gap: scan all edges with multiple `source_records` and upgrade confidence where warranted, regardless of whether a collapse happened

## TODO

### 1. Add a global confidence upgrade pass to `curator.py`

Run once per curator cycle, after the dedup passes and after the expiry scan.

The scan:

```
1. Query graph for all fact edges where:
   - len(source_records) > 1
   - confidence = 'inferred'
   - stale = false
2. For each edge:
   a. Fetch the confidence value as produced by each source_record in source_records
   b. If any source produced 'stated': set edge confidence = 'stated', log upgrade
   c. Otherwise: leave unchanged
```

No LLM needed — purely mechanical. Same upgrade logic as 1.14.2 and 1.14.3, applied globally.

**How to fetch per-source confidence:**
The confidence on the edge reflects the highest confidence at write time. To check per-source confidence, the curator needs access to the extraction output for each source record — specifically the `confidence` field on the extracted fact that produced this edge. This may require storing per-source confidence in `source_records` as a list of `{record_id, confidence}` objects rather than a flat list of strings.

If `source_records` is currently a flat list of strings, this spec requires changing the structure to `[{record_id, confidence}, ...]` — update `create_fact_edge` and the literal dup merge logic in the indexer accordingly. This is a schema change and must be reflected in `graph.py`.

If the per-source confidence is not available (legacy edges, migration gap), skip the upgrade for that edge and log a warning.

### 2. Write `docs/curator.md`

Create a new documentation file `docs/curator.md`. This is the canonical reference for the Curator agent — its purpose, behavior, and known limitations. Written for a developer reading the codebase for the first time.

Content:

**What the Curator is:**
A standalone async worker that runs after each record is indexed. Its job is to make the graph reality-aligned over time. It is not a ground-truth authority — it is a best-effort cleanup pass that improves graph quality without blocking the write path.

**What it does:**
- AFFILIATED semantic dedup: collapses semantically equivalent AFFILIATED edges across records. Stales the older one by `source_at`. Genuinely distinct roles coexist.
- ASSERTED semantic dedup: collapses semantically equivalent ASSERTED facts (same claim, different wording). Stales the redundant one. Genuinely distinct claims coexist.
- Expired commitment detection: stales ASSERTED[commitment] and ASSERTED[promise] facts where `valid_until` has passed with no subsequent resolution.
- Confidence upgrades: upgrades surviving edges from `inferred` to `stated` when merged source records include a source that explicitly stated the fact.

**How it runs:**
- Follows the same worker loop pattern as the indexer: poll `curator_queue`, claim one entry, process inline, delete entry, repeat
- One entry at a time — no concurrency
- The expiry scan and global confidence upgrade pass run once per cycle regardless of which record was claimed

**Triggering:**
The indexer enqueues `record_id` in `curator_queue` after `_mark_indexed` succeeds. The Curator never touches records where `indexed=false`.

**Crash recovery:**
Abandoned claims (where the Curator crashed mid-processing) are detected by timeout and reset on the next cycle. Default timeout: 10 minutes.

**Intended limitations:**
- The Curator is internal only — no external API calls, no HIL, no expert agents
- Semantic dedup uses an LLM judge: it may miss equivalences or produce false positives, especially on ASSERTED facts. False positives (wrongly collapsing distinct claims) are worse than false negatives — the judge prompt is calibrated accordingly.
- Expired commitment detection is mechanical: it stales on deadline expiry regardless of whether the commitment was actually fulfilled. A completed commitment without a TRANSITIONED[completion] edge will appear expired. The Augmentation agent (future) handles resolution confirmation.
- The `_notify_expiry` hook is a no-op — future HIL/agent notification is reserved but not yet implemented.
- Equal `source_at` conflicts are left unresolved — logged but not acted on. The Augmentation agent (future) resolves these with external corroboration.
- The Curator never deletes — it only sets `stale`, `replaced_by`, and `confidence`. History is always preserved.

## What NOT to do

- Do not run the global confidence upgrade before the dedup passes — run it last
- Do not upgrade edges that already have `confidence=verified` or `confidence=retracted`
- Do not skip the schema change if `source_records` is currently a flat list — per-source confidence is required for correctness
- Do not implement external enrichment or HIL here — that belongs to the Augmentation agent
- Do not remove the per-slot confidence upgrade from 1.14.2 and 1.14.3 — both are needed. The per-slot upgrade fires immediately after a collapse; the global pass catches edges that were never collapsed.
