# Context

- The graph has entities, relationships, and facts from emails and Linear issues. Change history from Linear is captured as regular facts (v1.6.3).
- Currently all facts and relationships are stored as-is — no notion of when they became true, when they stopped being true, or whether a newer fact supersedes an older one. "Status = In Progress" and "Status = In Review" both exist as separate facts with no temporal relationship between them.
- The bi-temporal model fixes this. Every edge gets timestamps that track real-world validity and when the system recorded it. Nothing is deleted — history is preserved, point-in-time queries become possible.

# TODO

## Add temporal timestamps to all graph edges

Every relationship and fact edge in Neo4j gets four timestamps:

- `valid_at` — when this became true in the real world
- `invalid_at` — when it stopped being true (null if still current)
- `created_at` — when PearScaff recorded it (already exists)
- `source_record` — which record told us this (already exists)

Relationships (works_at, customer_of, etc.) and facts (HAS_FACT edges to Fact nodes) both get the same temporal properties.

## Change fact storage from update-in-place to invalidate-and-create

Currently `upsert_fact` uses MERGE — if a fact with the same claim exists, it updates it. With bi-temporal modeling, instead of updating, it should:

1. Find the existing current fact (where `invalid_at` is null)
2. Set `invalid_at` to now on the old fact
3. Create a new fact with `valid_at` = now, `invalid_at` = null

This preserves the full history. "Status = In Progress" with `valid_at = March 10, invalid_at = March 15` followed by "Status = In Review" with `valid_at = March 15, invalid_at = null`.

The same logic applies to relationships — if "Michael works_at Acme" changes to "Michael works_at Beta Corp", the old relationship gets invalidated, the new one is created.

## Retriever queries become temporal-aware

The Retriever's tools should default to querying current state (`invalid_at IS NULL`) but support point-in-time queries ("what was the status of the Acme deal on March 10?"). The `facts_lookup` and `graph_traverse` tools should filter to current facts by default.

## Retrofit existing data

Existing edges and facts that don't have temporal timestamps get `valid_at = created_at` and `invalid_at = null` (assumed still current). This is a one-time migration on the Neo4j data.

## Memory CLI temporal display

`ps memory entity` and `ps memory record` should show temporal information — when facts became valid, whether they've been superseded. Superseded facts could be shown dimmed or with a "[was]" marker.

## What NOT to do

- No new data sources or extraction changes
- No changes to how the Indexer calls extraction — only how `graph.py` stores the results
- No changes to Linear or Gmail polling
- No event sourcing on Postgres tables — temporal modeling is Neo4j only for now
