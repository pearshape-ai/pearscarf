# Plan: 1.1.3 ‚Äî Mem0 OpenAI Provider + Data Consolidation

## Summary

Two changes in this version:

1. **Mem0 LLM provider switch**: Mem0's Anthropic provider is poorly maintained ‚Äî sends both `temperature` and `top_p`, passes OpenAI-style `tool_choice` strings, and uses OpenAI-format tool definitions. Rather than patching each incompatibility, switch to OpenAI (Mem0's native provider). Anthropic remains the LLM for all PearScaff agents.

2. **Data consolidation**: All persistent data (SQLite, ChromaDB, logs, Neo4j, Qdrant, browser state) moved under a single `data/` directory in the repo root, gitignored. Docker containers bind-mount into `data/` for local persistence. Qdrant switched from local file-based to Docker server to support concurrent access from multiple processes.

---

## Context

### Mem0 Anthropic Issues

Mem0 was built for OpenAI. The Anthropic provider has multiple API bugs:

1. **temperature + top_p**: Sends both; Anthropic rejects the combo
2. **tool_choice format**: Passes `"auto"` (string); Anthropic needs `{"type": "auto"}`
3. **tool definitions**: Uses OpenAI format `{"type": "function", "function": {...}}`; Anthropic expects `{"name": ..., "input_schema": ...}`

### Data Scattered Across Root

Before this change, data files were spread across the repo root:
- `pearscaff.db`, `pearscaff.db-shm`, `pearscaff.db-wal`
- `chroma_data/`
- `logs/`
- `storage_state.json`
- Docker volumes for Neo4j and Qdrant were either unnamed or didn't exist

### Qdrant Locking

Local file-based Qdrant uses SQLite internally and can't be opened by two processes simultaneously. Running `ps discord` (indexer writes) and `ps memory list` (reads) at the same time causes `attempt to write a readonly database`. Switching to a Qdrant Docker server solves this.

## Changes

### 1. Switch Mem0 LLM config to OpenAI

**File:** `pearscaff/memory.py`

```python
"llm": {
    "provider": "openai",
    "config": {
        "model": OPENAI_MODEL,
        "api_key": OPENAI_API_KEY,
    },
},
```

### 2. Remove Anthropic compatibility patches

**File:** `pearscaff/memory.py`

Removed `_patch_mem0_llm()` static method and all monkey-patching of `generate_response` and `_get_common_params`.

### 3. Switch Qdrant to server mode

**File:** `pearscaff/memory.py`

Replaced `path: QDRANT_PATH` (local file) with `url: QDRANT_URL` (server). Default: `http://localhost:6333`.

### 4. Fix Qdrant exit traceback

**File:** `pearscaff/memory.py`

Patch `QdrantClient.__del__` to no-op at the class level. Explicit `close()` in atexit handler.

### 5. Add OpenAI + Qdrant config vars

**File:** `pearscaff/config.py`

```python
OPENAI_API_KEY = os.getenv("OPENAI_API_KEY", "")
OPENAI_MODEL = os.getenv("OPENAI_MODEL", "gpt-4o-mini")
QDRANT_URL = os.getenv("QDRANT_URL", "http://localhost:6333")
```

### 6. Consolidate all data under `data/`

| Path | What | Config |
|------|------|--------|
| `data/pearscaff.db` | SQLite (sessions, messages, records, graph) | `DB_PATH` |
| `data/chroma/` | ChromaDB vector store (sqlite backend) | `CHROMA_PATH` |
| `data/logs/session.log` | Unified agent log | Hardcoded in `log.py` |

## Architecture Note

PearScaff now uses two LLM providers:
- **Anthropic** (Claude) ‚Äî all PearScaff agents (worker, experts, retriever), SQLite backend extraction
- **OpenAI** (gpt-4o-mini) ‚Äî Mem0 internal extraction only (memory.add calls)

All persistent data lives under `data/` (gitignored). Docker containers bind-mount into it for Neo4j and Qdrant.
