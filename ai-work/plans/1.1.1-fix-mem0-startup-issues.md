# Plan: 1.1.1 — Fix Mem0 Startup Issues

Issues discovered during first Mem0 integration run.

---

## Fixes

### 1. Mem0 defaulted to OpenAI embeddings

Mem0 requires an explicit embedder config — without it, it tries to use OpenAI and fails with `OPENAI_API_KEY` not set. Fixed by adding HuggingFace embedder config with `all-MiniLM-L6-v2` (same model already used by ChromaDB) and `embedding_dims: 384`.

**File:** `pearscaff/memory.py`

### 2. Neo4j auth failure from `#` and `$` in password

`python-dotenv` treats unquoted `#` as inline comments and the shell expands `$` as variable references. Passwords with special characters must be quoted in `.env`. Updated docs to use `$NEO4J_PASSWORD` from env instead of hardcoded `password` in the docker command.

**File:** `docs/getting-started.md`

### 3. Lazy imports for ChromaDB and graph modules

When `MEMORY_BACKEND=mem0`, ChromaDB and the SQLite graph module were still imported at module level, causing unnecessary model loads and slow startup. Moved `vectorstore` and `graph` imports to be lazy (inside `execute()` methods) so they only load when `MEMORY_BACKEND=sqlite`.

**Files:** `pearscaff/memory.py`, `pearscaff/experts/retriever.py`

### 4. Clean Mem0 startup messaging

Instead of suppressing Mem0's progress bar output (which hides real errors), added "Initializing Mem0..." / "Mem0 ready." messages to frame the model loading output as intentional.

**File:** `pearscaff/memory.py`

### 5. Qdrant cleanup on exit

Mem0 uses Qdrant internally. On Ctrl+C, Python's garbage collector tries to close the Qdrant client after the import system is torn down, producing noisy `ImportError` tracebacks. Added an `atexit` handler to close the Qdrant client cleanly before shutdown.

**File:** `pearscaff/memory.py`
