# Plan: 1.1.0 — LangSmith Observability

Spec: `ai-work/specs/1.1.0-langsmith.md`

## Summary

Add LangSmith tracing to every LLM call in the system. Every agent turn, tool execution, and memory operation becomes a traced span with token counts, latency, and cost. Tracing is opt-in — when not configured, the system runs exactly as before.

|----------|----------|-------|-------|

| `memory.py:297` | `SqliteBackend._extract()` | indexer | Single extraction call |
| Mem0 internal | `mem0.add()` / `mem0.search()` | indexer/retriever | Opaque — wrap as parent span |

### Step 2: Create `pearscaff/tracing.py` — tracing utilities

**New file.** Thin wrapper around LangSmith's `@traceable` decorator and `RunTree` context.

```python
from pearscaff.config import LANGSMITH_ENABLED

def traceable(name: str, run_type: str = "llm", metadata: dict | None = None):
    """Decorator that traces a function if LangSmith is enabled, no-op otherwise."""
    if not LANGSMITH_ENABLED:
        def noop(fn):
            return fn
        return noop

    from langsmith import traceable as ls_traceable
    return ls_traceable(name=name, run_type=run_type, metadata=metadata or {})


def trace_context(name: str, run_type: str = "chain", metadata: dict | None = None):
    """Context manager for tracing a block. No-op when disabled."""
    if not LANGSMITH_ENABLED:
        from contextlib import nullcontext
        return nullcontext()

    from langsmith.run_trees import RunTree
    return RunTree(name=name, run_type=run_type, extra={"metadata": metadata or {}})
```

**Key decisions:**
- When `LANGSMITH_ENABLED=false`, decorators and context managers are no-ops — zero overhead
- Lazy import of `langsmith` — only loaded when tracing is on
- Centralizes all tracing logic — agents don't import langsmith directly

### Step 3: Trace `BaseAgent.run()` — the main agent loop

**File:** `pearscaff/agents/base.py`

This is the most important trace — every agent conversation turn.

**Approach:** Wrap the `messages.create()` call and the full `run()` method.

1. Add an `agent_name` parameter to `BaseAgent.__init__()` (default `"agent"`)
2. Wrap `run()` to create a parent trace span per invocation
3. Wrap each `messages.create()` call as a child LLM span
4. Wrap each tool execution as a child tool span
5. Tag traces with: `agent_name`, `model`, `turn_number`

```python
def run(self, user_message: str) -> str:
    # Parent span: "worker.run" / "gmail_expert.run" etc.
    with trace_agent_run(self._agent_name, user_message) as run_ctx:
        # ... existing loop ...
        # Each messages.create() tagged as LLM child span
        # Each tool.execute() tagged as tool child span
```

**What the trace looks like:**
```
worker.run (chain)
  ├── llm_call_1 (llm) — tokens: 1200 in, 400 out
  ├── send_message (tool) — delegated to gmail_expert
  ├── llm_call_2 (llm) — tokens: 800 in, 200 out
  └── send_message (tool) — reply to human
```

### Step 5: Trace Mem0 calls as parent spans

**File:** `pearscaff/memory.py`

Mem0's internal LLM calls are opaque. We can't instrument them individually without modifying Mem0's internals. Instead:

1. Wrap `Mem0Backend.add()` as a `"mem0.add"` chain span
2. Wrap `Mem0Backend.search()` as a `"mem0.search"` chain span
3. If Mem0 supports passing a custom LLM client: pass a LangSmith-wrapped Anthropic client via Mem0's config. This makes Mem0's internal calls visible as child spans.
4. If not feasible, the parent spans still give us latency and we can infer cost from the token usage in Anthropic's billing.

### Step 6: Trace the Indexer processing loop

**File:** `pearscaff/indexer.py`

Wrap `_process_record()` as a chain span:
- name: `"indexer.process_record"`
- metadata: `record_id`, `record_type`
- Children: the `memory.add()` call from step 4/5

### Step 7: Trace the AgentRunner message dispatch

**File:** `pearscaff/agents/runner.py`

Wrap `_process_message()` as the top-level chain span:
- name: `"{agent_name}.process_message"`
- metadata: `session_id`, `from_agent`, `agent_name`
- Children: the `agent.run()` call from step 3

This creates the hierarchical trace structure:
```
worker.process_message (chain)
  └── worker.run (chain)
        ├── llm_call (llm)
        ├── tool: send_message (tool)
        └── llm_call (llm)
```

## Trace Hierarchy (full picture)

```
worker.process_message (chain) [session_id, from_agent]
  └── worker.run (chain) [agent=worker]
        ├── llm_call_1 (llm) [model, tokens_in, tokens_out]
        ├── tool: classify_record (tool)
        ├── llm_call_2 (llm)
        └── tool: send_message → gmail_expert (tool)

gmail_expert.process_message (chain) [session_id]
  └── gmail_expert.run (chain) [agent=gmail_expert]
        ├── llm_call_1 (llm)
        ├── tool: gmail_get_unread (tool)
        ├── llm_call_2 (llm)
        ├── tool: save_email (tool)
        └── tool: reply (tool)

indexer.process_record (chain) [record_id]
  └── mem0.add (chain) OR indexer.extract (llm)
        └── [Mem0 internal LLM calls if instrumented]

retriever.process_message (chain) [session_id]
  └── retriever.run (chain)
        ├── llm_call_1 (llm)
        ├── tool: memory_search (tool)
        │     └── mem0.search (chain)
        └── tool: reply (tool)
```
