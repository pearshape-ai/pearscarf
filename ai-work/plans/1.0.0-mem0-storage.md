# Plan: 1.0.0 — Mem0 Storage Integration

Spec: `ai-work/specs/1.0.0-mem0-storage.md`

## Summary

Replace the hand-rolled facts table, SQLite graph, and ChromaDB with Mem0 (backed by Neo4j). The Indexer shrinks to a thin wrapper around `mem0.add()`. The Retriever unifies three separate queries into one `mem0.search()` call. A `MEMORY_BACKEND` env var switches between `mem0` and the existing `sqlite` pipeline.

### Step 2: Create `pearscaff/memory.py` — pluggable memory interface

**New file.** Defines the memory backend abstraction and both implementations.

```python
# Abstract interface
class MemoryBackend:
    def add(self, content: str, metadata: dict, user_id: str = "default") -> None: ...
    def search(self, query: str, user_id: str = "default", limit: int = 10) -> list[dict]: ...

# Mem0 implementation
class Mem0Backend(MemoryBackend):
    def __init__(self):
        from mem0 import Memory
        self._mem = Memory.from_config({
            "graph_store": {
                "provider": "neo4j",
                "config": {
                    "url": NEO4J_URL,
                    "username": NEO4J_USER,
                    "password": NEO4J_PASSWORD,
                }
            },
            "llm": {
                "provider": "anthropic",
                "config": {"model": MODEL, "api_key": ANTHROPIC_API_KEY}
            },
            "custom_prompt": CUSTOM_EXTRACTION_PROMPT,  # see step 3
        })

    def add(self, content, metadata, user_id="default"):
        self._mem.add(content, user_id=user_id, metadata=metadata)

    def search(self, query, user_id="default", limit=10):
        return self._mem.search(query, user_id=user_id, limit=limit)

# SQLite fallback (wraps existing graph.py + vectorstore.py)
class SqliteBackend(MemoryBackend):
    # Delegates to existing graph + vectorstore code
    # Keeps current behavior when MEMORY_BACKEND=sqlite

# Factory
def get_memory_backend() -> MemoryBackend:
    if MEMORY_BACKEND == "mem0":
        return Mem0Backend()
    return SqliteBackend()
```

**Key decisions:**
- Lazy singleton — initialized on first use, not at import
- `user_id` defaults to `"default"` (single-user for now)
- `SqliteBackend.add()` replicates current indexer logic (extract → entities → edges → facts → embed)
- `SqliteBackend.search()` replicates current retriever logic (facts + graph + vector)

### Step 3: Custom extraction prompt for Mem0

**In `pearscaff/memory.py`** — constant string passed to Mem0 config.

Guide Mem0 toward extracting:
- People (names, emails, roles, titles)
- Companies (names, domains, industries)
- Projects and initiatives
- Financial details (amounts, invoices, budgets)
- Commitments and deadlines
- Relationships between entities

Explicitly tell it to ignore:
- Greetings, signatures, disclaimers
- Timestamps that are just "when the email was sent"
- Boilerplate footer text

### Step 4: Modify `pearscaff/indexer.py`

**Current pipeline (6 stages):** poll → extract via LLM → resolve entities → create edges → upsert facts → embed in ChromaDB → mark indexed

**New pipeline with Mem0:**
1. Poll unindexed + classified=relevant records (unchanged)
2. Build content string from email (`_build_content` — reuse existing)
3. Call `memory_backend.add(content, metadata)` where metadata includes `record_id`, `sender`, `subject`, `type`
4. Mark indexed (unchanged)

**What changes:**
- `_extract()` — no longer called when backend is mem0
- `_resolve_entity()` — no longer called when backend is mem0
- Graph population loop — no longer called when backend is mem0
- Vector embedding — no longer called when backend is mem0

**Implementation approach:**
```python
def _process_record(self, record):
    content = self._build_content(record)
    metadata = {"record_id": record["id"], "type": record["type"],
                "sender": record.get("sender"), "subject": record.get("subject")}

    if record.get("human_context"):
        content += f"\n\nHuman annotation: {record['human_context']}"

    self._memory.add(content, metadata)
    self._mark_indexed(record["id"])
```

The `SqliteBackend.add()` keeps the old extraction pipeline for fallback.

### Step 5: Modify `pearscaff/experts/retriever.py`

**Current tools (4):** `search_entities`, `facts_lookup`, `graph_traverse`, `vector_search`

**New approach with Mem0:**
- Replace all four tools with a single `MemorySearchTool`
- The tool calls `memory_backend.search(query)` and formats results
- Response format stays the same context package shape so the worker doesn't notice

**When MEMORY_BACKEND=sqlite:** register the existing four tools (no change)
**When MEMORY_BACKEND=mem0:** register `MemorySearchTool` only

```python
class MemorySearchTool(BaseTool):
    name = "memory_search"
    description = "Search memories for context about a topic, person, or company"
    input_schema = {
        "type": "object",
        "properties": {
            "query": {"type": "string", "description": "What to search for"}
        },
        "required": ["query"]
    }

    def execute(self, **kwargs):
        results = self._memory.search(kwargs["query"])
        # Format into context package
```

**Update retriever system prompt** to reflect new tool when mem0 is active:
- Old: "You have four tools: search_entities, facts_lookup, graph_traverse, vector_search"
- New: "You have one tool: memory_search — it searches across memories and graph connections"
