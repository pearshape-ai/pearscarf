# Plan: 1.6.2 — Linear Extraction

Spec: `ai-work/specs/1.6.2-linear-extraction.md`

## Overview

Issues classified as "relevant" already flow through the Indexer's `_loop` → `_process_record` → `_build_content` → `_extract` → Neo4j → Qdrant pipeline with no code gating on record type. The `_build_content` method already handles `type == "issue"`. The main work is tuning the extraction prompt so it handles issue-shaped content well, and verifying cross-source entity resolution.

**No structural code changes to the pipeline.** This is a prompt change + verification pass.

## File Changes

### 1. `pearscaff/prompts/extraction.md` — Make source-agnostic, add issue guidance

The current prompt opens with "operational business emails" and has email-specific language scattered throughout. Issues don't have senders, recipients, signatures, or CC lists. The prompt needs to:

1. Broaden the opening line from "emails" to "records" (emails and issues)
2. Add a short issue-specific section so the LLM knows what to focus on
3. Keep all existing email guidance (signature blocks, CC lists, footers) — it's still correct for emails and harmlessly ignored for issues

**Changes:**

Line 1 — broaden opening:
```
Before: You are an entity extraction system for operational business emails.
After:  You are an entity extraction system for operational business records (emails and issues).
```

Line 5 — broaden normalization intro:
```
Before: regardless of how it appears across different emails.
After:  regardless of how it appears across different records.
```

Person entity (line 19) — broaden extraction triggers:
```
Before: Extract when: they are the sender, a direct recipient, or someone actively discussed in the email
After:  Extract when: they are the sender, a direct recipient, the assignee, or someone actively discussed in the record
```

Add a new section after "## What to ignore" (before "## Output format") for issue-specific guidance:

```markdown
## Issue-specific guidance

When the record is an issue (indicated by "Issue:" prefix in the content):

- **Don't re-extract structured fields.** The assignee, status, priority, project, and labels are already stored as structured data. Focus extraction on the description and comments — that's where unstructured knowledge lives.
- **Extract people mentioned in comments.** Comments often reference people by first name or @-mention. Extract them as person entities when they're actively involved ("@Sarah can you review this?" → person Sarah with relationship to the project).
- **Extract commitments and blockers from comments.** "Blocked on Acme's API key" → fact on the project entity. "Pushing to next sprint" → fact about timeline.
- **Extract project references.** Issues often reference other projects or initiatives in description/comments. These cross-references are high-value for the graph.
```

Update "stated" confidence description:
```
Before: stated — explicitly said in the email
After:  stated — explicitly said in the record
```

### 2. No other file changes

The Indexer, graph module, store, and expert code require zero changes:

- **`indexer.py`**: `_loop` already queries `WHERE indexed = FALSE AND classification = 'relevant'` with no type filter. `_build_content` already handles `type == "issue"`. `_process_record` is fully generic. `_embed_record` already handles issue metadata.
- **`graph.py`**: `find_entity` and `create_entity` use name + email/domain matching with MERGE. This handles cross-source resolution — a person "Michael Chen" from an email and the same name from an issue comment will resolve to the same node.
- **`store.py`**: Issues with `classification = 'relevant'` and `indexed = FALSE` will be picked up. No changes needed.
- **`experts/linear.py`**: No changes to tools or polling per spec.

## Verification

### Pipeline flow test

1. Ensure at least one issue is classified as `'relevant'` in the database:
   ```sql
   SELECT id, type, classification, indexed FROM records WHERE type = 'issue';
   ```
2. If none are relevant yet, manually classify one:
   ```sql
   UPDATE records SET classification = 'relevant' WHERE id = 'issue_001';
   ```
3. Start the system — the Indexer should pick up the issue within 5 seconds
4. Check logs for: `[indexer] processing issue_001`
5. Check logs for entity/relationship/fact counts: `indexed issue_001: N entities, M relationships, K facts`

### Cross-source entity resolution test

1. Find an entity that exists from email extraction:
   ```sql
   -- In Neo4j:
   MATCH (n:Person) RETURN n.name, n.email LIMIT 10
   ```
2. Find an issue that mentions the same person (by name in description or comments, or as assignee)
3. After the issue is indexed, verify the same Neo4j node was reused (not a duplicate):
   ```sql
   MATCH (n:Person {name: "Michael Chen"}) RETURN count(n)
   -- Should be 1, not 2
   ```
4. Check that the entity now has edges from both the email source record and the issue source record

### Edge cases to watch

- **Assignee-only issues**: If an issue has an assignee but empty description and no comments, the extraction should still pick up the assignee as a person entity (from the `_build_content` output: `Assignee: Michael Chen`)
- **Name normalization across sources**: Email might say "Mike Chen" while the issue assignee field says "Michael Chen". The extraction prompt's normalization rules should produce the same name, but if Linear uses a different form than email, `find_entity` won't match on name alone — it would create a duplicate. This is a known limitation of name-based matching.
- **Issue with no description**: Some issues are title-only. The extraction should still work — `_build_content` handles this gracefully by skipping empty fields.
