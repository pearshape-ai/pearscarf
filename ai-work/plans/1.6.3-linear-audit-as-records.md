# Plan: 1.6.3 — Linear Audit as Records

Spec: `ai-work/specs/1.6.3-linear-audit-as-records.md`

## Overview

Capture meaningful issue change history (status, assignee, priority transitions) from Linear's `issueHistory` API, store each change as its own record in the SOR, and feed them through the existing extraction pipeline. No bi-temporal timestamps — facts accumulate as regular Fact nodes in Neo4j.

---

## Linear's IssueHistory API

Each issue exposes a `history` connection with entries containing:
- `id`, `createdAt`, `actor { name email }`
- `fromState { name }` / `toState { name }` — status changes
- `fromAssignee { name }` / `toAssignee { name }` — reassignments
- `fromPriority` / `toPriority` — priority changes (int 0-4)
- `fromProject { name }` / `toProject { name }` — project changes
- `fromEstimate` / `toEstimate`, `fromDueDate` / `toDueDate`, etc.

We only capture **status, assignee, and priority** changes per spec (the most meaningful transitions).

## File Changes

### 1. `pearscaff/linear_client.py` — Add `get_issue_history` method

New method to fetch history entries for a specific issue. Uses cursor pagination like existing methods.

```python
def get_issue_history(
    self, issue_id: str, since: str | None = None
) -> list[dict]:
    """Fetch meaningful history entries (status, assignee, priority changes) for an issue.

    If `since` is provided (ISO timestamp), only returns entries after that time.
    Returns list of dicts with: id, created_at, actor, field, from_value, to_value.
    """
    all_entries: list[dict] = []
    cursor: str | None = None
    while True:
        vars_: dict = {"issueId": issue_id, "first": 50}
        if cursor:
            vars_["after"] = cursor

        data = self._query("""
            query($issueId: String!, $first: Int, $after: String) {
                issue(id: $issueId) {
                    history(first: $first, after: $after) {
                        nodes {
                            id
                            createdAt
                            actor { name email }
                            fromState { name }
                            toState { name }
                            fromAssignee { name }
                            toAssignee { name }
                            fromPriority
                            toPriority
                        }
                        pageInfo { hasNextPage endCursor }
                    }
                }
            }
        """, variables=vars_)

        issue_data = data.get("issue", {})
        history = issue_data.get("history", {})
        nodes = history.get("nodes", [])
        page_info = history.get("pageInfo", {})

        for node in nodes:
            created_at = node.get("createdAt", "")
            if since and created_at <= since:
                continue

            actor_name = (node.get("actor") or {}).get("name", "")

            # Status change
            if node.get("fromState") or node.get("toState"):
                all_entries.append({
                    "id": node["id"],
                    "created_at": created_at,
                    "actor": actor_name,
                    "field": "status",
                    "from_value": (node.get("fromState") or {}).get("name", ""),
                    "to_value": (node.get("toState") or {}).get("name", ""),
                })

            # Assignee change
            if node.get("fromAssignee") or node.get("toAssignee"):
                all_entries.append({
                    "id": node["id"] + "_assignee",
                    "created_at": created_at,
                    "actor": actor_name,
                    "field": "assignee",
                    "from_value": (node.get("fromAssignee") or {}).get("name", ""),
                    "to_value": (node.get("toAssignee") or {}).get("name", ""),
                })

            # Priority change
            if node.get("fromPriority") is not None or node.get("toPriority") is not None:
                priority_labels = {0: "No priority", 1: "Urgent", 2: "High", 3: "Medium", 4: "Low"}
                from_p = node.get("fromPriority")
                to_p = node.get("toPriority")
                if from_p != to_p:
                    all_entries.append({
                        "id": node["id"] + "_priority",
                        "created_at": created_at,
                        "actor": actor_name,
                        "field": "priority",
                        "from_value": priority_labels.get(from_p, str(from_p)) if from_p is not None else "",
                        "to_value": priority_labels.get(to_p, str(to_p)) if to_p is not None else "",
                    })

        if not page_info.get("hasNextPage"):
            break
        cursor = page_info.get("endCursor")
        if not cursor:
            break

    return all_entries
```

A single history node can contain multiple changes (e.g. status AND priority changed at the same time). Each change becomes a separate entry with a suffixed ID for dedup (`node_id + "_assignee"`, `node_id + "_priority"`). Status changes use the base `node_id` since it's the most common.

### 2. `pearscaff/db.py` — Add `issue_changes` table

New table following the same pattern as `emails` and `issues` — a type-specific table that references `records(id)`:

```sql
CREATE TABLE IF NOT EXISTS issue_changes (
    record_id TEXT PRIMARY KEY REFERENCES records(id),
    issue_record_id TEXT NOT NULL,
    linear_history_id TEXT UNIQUE,
    field TEXT NOT NULL,
    from_value TEXT,
    to_value TEXT,
    changed_by TEXT,
    changed_at TIMESTAMPTZ NOT NULL
);

CREATE INDEX IF NOT EXISTS idx_issue_changes_issue ON issue_changes(issue_record_id);
CREATE INDEX IF NOT EXISTS idx_issue_changes_linear_id ON issue_changes(linear_history_id);
```

- `record_id` — the change's own record in the SOR (e.g. `change_001`)
- `issue_record_id` — links to the parent issue's record_id (e.g. `issue_001`)
- `linear_history_id` — Linear's history entry ID, used for dedup (UNIQUE)
- `field` — what changed: `status`, `assignee`, or `priority`
- `from_value`, `to_value` — human-readable old and new values
- `changed_by` — name of the person who made the change
- `changed_at` — when the change happened (Linear's timestamp)

No FK from `issue_record_id` to `issues(record_id)` — keep it loose in case the parent issue was saved before the schema existed.

Add the DDL to `_SCHEMA` after the issues table and its indexes.

### 3. `pearscaff/store.py` — Add `save_issue_change` and helpers

**New ID generator:**

```python
def _next_change_id() -> str:
    with _get_conn() as conn:
        row = conn.execute("SELECT COUNT(*) as c FROM records WHERE type = 'issue_change'").fetchone()
        num = row["c"] + 1
        return f"change_{num:03d}"
```

**New save function:**

```python
def save_issue_change(
    issue_record_id: str,
    field: str,
    from_value: str = "",
    to_value: str = "",
    linear_history_id: str | None = None,
    changed_by: str = "",
    changed_at: str = "",
) -> str | None:
    """Save an issue change to the SOR.

    Returns record_id (e.g. 'change_001'), or None if duplicate (same linear_history_id).
    Auto-classified as 'relevant' since the parent issue was already triaged.
    """
    init_db()
    with _get_conn() as conn:
        # Dedup on linear_history_id
        if linear_history_id:
            existing = conn.execute(
                "SELECT record_id FROM issue_changes WHERE linear_history_id = %s",
                (linear_history_id,),
            ).fetchone()
            if existing:
                return None

        record_id = _next_change_id()
        now = _now()

        conn.execute(
            "INSERT INTO records (id, type, source, created_at, classification) "
            "VALUES (%s, %s, %s, %s, %s)",
            (record_id, "issue_change", "linear_expert", now, "relevant"),
        )
        conn.execute(
            "INSERT INTO issue_changes (record_id, issue_record_id, linear_history_id, "
            "field, from_value, to_value, changed_by, changed_at) "
            "VALUES (%s, %s, %s, %s, %s, %s, %s, %s)",
            (record_id, issue_record_id, linear_history_id, field,
             from_value, to_value, changed_by, changed_at or now),
        )
        conn.commit()
        return record_id
```

Key design decisions:
- **Auto-classified as `'relevant'`** — parent issue was already triaged, so changes inherit relevance. This means the Indexer picks them up immediately.
- **Dedup on `linear_history_id`** — same as emails dedup on `message_id` and issues on `linear_id`.
- **Source is `"linear_expert"`** — matches the issue records.

### 4. `pearscaff/experts/linear.py` — Fetch changes during incremental polls

Update the polling loop to fetch history for updated issues. Only during **incremental** polls (not the initial bulk load — fetching history for hundreds of issues on first run would be too expensive).

Add a module-level helper:

```python
def _sync_issue_changes(
    client: LinearClient,
    issue: dict,
    issue_record_id: str,
    since: str | None,
) -> int:
    """Fetch and save history changes for an issue. Returns count of new changes saved."""
    from pearscaff import store

    changes = client.get_issue_history(issue["id"], since=since)
    saved = 0
    for change in changes:
        record_id = store.save_issue_change(
            issue_record_id=issue_record_id,
            field=change["field"],
            from_value=change.get("from_value", ""),
            to_value=change.get("to_value", ""),
            linear_history_id=change.get("id"),
            changed_by=change.get("actor", ""),
            changed_at=change.get("created_at", ""),
        )
        if record_id:
            saved += 1
    return saved
```

Update the incremental path in `_poll_loop`:

```python
# In the incremental (synced_at is not None) branch, after _save_issue_from_poll:
for issue in issues:
    record_id, is_new = _save_issue_from_poll(issue)

    # Fetch history changes for updated issues
    try:
        n_changes = _sync_issue_changes(client, issue, record_id, since=synced_at)
        if n_changes:
            log.write(
                "linear_expert", "--", "action",
                f"Poll: {n_changes} change(s) for {issue.get('identifier', '')}",
            )
    except Exception as exc:
        log.write("linear_expert", "--", "error",
                  f"History fetch failed for {issue.get('identifier', '')}: {exc}")

    if is_new:
        # ... existing session creation for new issues
```

The `since=synced_at` filter avoids re-fetching old history on every poll. Dedup via `linear_history_id` provides a second safety net.

**No changes to the initial bulk load path** — it already does batch triage, and fetching history for all issues on first run is too expensive. History will be captured from the first incremental poll onward.

## Change-specific guidance

When the record is an issue change (indicated by "Change:" prefix in the content):

- **Extract the transition as a fact.** A status change from "In Progress" to "In Review" on "Acme API Integration" → fact: "Moved to In Review" on the project entity. A priority escalation from "Medium" to "Urgent" → fact: "Priority escalated to Urgent".

## Data Flow

```
Linear API (issueHistory)
    ↓ polling loop (incremental only)
store.save_issue_change()
    ↓ records table (type='issue_change', classification='relevant')
    ↓ issue_changes table (field, from/to, changed_by, changed_at)
Indexer polls (classification='relevant', indexed=FALSE)
    ↓ _build_content (joins issue_changes + issues for context)
    ↓ _extract (LLM produces facts about transitions)
    ↓ entity resolution (reuses existing entities from parent issue)
Neo4j (Fact nodes: "Moved to In Review", "Priority escalated to Urgent")
Qdrant (embedded change record with metadata)
    ↓ mark indexed
```
