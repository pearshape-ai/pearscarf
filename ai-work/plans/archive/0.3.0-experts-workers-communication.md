# Plan 0.3.0: Agent Communication System
**Date:** 2026-02-27
**Status:** Draft

## Context

Agents are currently standalone — `WorkerAgent` runs directly in the REPL/Discord, `ExpertAgent` runs in its own REPL. This version adds async agent-to-agent communication via SQLite, sessions, a session-aware REPL, and Discord thread-per-session mapping.

## Database Schema (`db.py`)

```sql
CREATE TABLE sessions (
    id TEXT PRIMARY KEY,          -- "ses_001", "ses_002", ...
    initiated_by TEXT NOT NULL,   -- "human", "gmail_expert", etc.
    summary TEXT NOT NULL DEFAULT '',
    created_at TEXT NOT NULL
);

CREATE TABLE messages (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id TEXT NOT NULL REFERENCES sessions(id),
    from_agent TEXT NOT NULL,     -- "human", "worker", "gmail_expert"
    to_agent TEXT NOT NULL,
    content TEXT NOT NULL,
    reasoning TEXT NOT NULL DEFAULT '',
    data TEXT NOT NULL DEFAULT '{}',   -- flexible JSON
    read INTEGER NOT NULL DEFAULT 0,
    created_at TEXT NOT NULL
);

CREATE INDEX idx_messages_to_unread ON messages(to_agent, read);
CREATE INDEX idx_messages_session ON messages(session_id);
```

`db.py` provides:
- `init_db()` — create tables if not exist, WAL mode
- `create_session(initiated_by, summary) -> session_id` — auto-generates sequential IDs
- `list_sessions() -> list[dict]`
- `insert_message(session_id, from_agent, to_agent, content, reasoning, data) -> msg_id`
- `poll_unread(to_agent) -> list[dict]` — returns unread messages for an agent, ordered by created_at
- `mark_read(msg_id)`
- `get_history(session_id) -> list[dict]`

## Message Bus (`bus.py`)

Thin wrapper over `db.py` for the common send/receive pattern:

```python
class MessageBus:
    def send(self, session_id, from_agent, to_agent, content, reasoning="", data=None)
    def poll(self, agent_name) -> list[dict]   # returns unread messages + marks them read
    def create_session(self, initiated_by, summary) -> str
    def list_sessions(self) -> list[dict]
    def get_history(self, session_id) -> list[dict]
```

## Agent Runner (`agents/runner.py`)

Each agent type gets a runner that:
1. Polls the bus every 1 second for unread messages addressed to it
2. Groups messages by session_id
3. For each message: rebuilds the agent's conversation context from session history, calls `agent.run(message_content)`, sends the response back via the bus

```python
class AgentRunner:
    def __init__(self, agent_name, agent_factory, bus):
        # agent_factory creates a fresh agent per session or returns cached one

    def run_loop(self):
        while not self._stop:
            messages = self.bus.poll(self.agent_name)
            for msg in messages:
                response = self._process(msg)
                self.bus.send(msg.session_id, self.agent_name, msg.from_agent, response, reasoning=...)
            time.sleep(1)
```

### Worker Runner

The worker agent gets a system prompt that knows about available experts and includes a `delegate_to_expert` tool:
- Input: `expert_name`, `task`, `session_id`
- Sends a message via the bus to the expert
- Returns "Delegated to {expert}. Waiting for response."

When the worker receives a response from an expert (message from expert to worker), it processes it and responds to human.

### Expert Runner

The expert agent (e.g. Gmail) runs with its browser tools. When it receives a message from the worker, it processes it using the browser and responds back.

Experts can also initiate sessions: e.g. if detecting an urgent email during a periodic check, the expert creates a new session and sends a message to the worker.

## REPL (`repl.py`)

Replaces the inline REPL in `cli.py`. Session-aware with commands:

```
[ses_001] > Read my latest emails
[ses_001] > /sessions
[ses_001] > /switch ses_002
[ses_002] > /history
[ses_002] > /new
[ses_003] >
```

Implementation:
- Main thread writes human messages to bus, polls for worker responses to human
- Polling runs in a background thread, printing responses and notifications as they arrive
- Notifications for expert-initiated sessions interrupt the prompt:
  ```
  --- NEW SESSION ses_003: gmail_expert — urgent email from investor ---
  ```

Commands:
- `/sessions` — list all sessions (id, initiated_by, summary)
- `/switch <id>` — switch active session
- `/new` — create a new session
- `/history` — print all messages in current session
- `/history <id>` — print messages in a specific session

## Discord Bot Refactor

- Each session maps 1:1 to a Discord thread
- New message in main channel → creates new session + new Discord thread
- Follow-up messages in a thread are tagged with that thread's session ID
- Expert-initiated sessions auto-create a thread in a designated channel
- Sessions are never closed; posting in a thread resumes the session

Key mappings stored in a new SQLite table:
```sql
CREATE TABLE discord_threads (
    session_id TEXT PRIMARY KEY REFERENCES sessions(id),
    thread_id INTEGER NOT NULL,
    channel_id INTEGER NOT NULL
);
```

## CLI (`cli.py`)

```
pearscaff run                     # starts worker + experts + REPL (all in one process)
pearscaff discord                 # starts worker + experts + Discord bot
pearscaff expert gmail --login    # login flow (unchanged)
pearscaff chat                    # DEPRECATED or kept as simple direct mode
```

`pearscaff run`:
1. `init_db()`
2. Start Gmail expert runner in a background thread
3. Start worker runner in a background thread
4. Run REPL in main thread

`pearscaff discord`:
1. `init_db()`
2. Start Gmail expert runner in a background thread
3. Start worker runner in a background thread
4. Run Discord bot in main thread (async event loop)

## Message Flow Examples

### Human asks to read emails
```
human → worker (ses_001): "Read my latest unread emails"
worker reasons: "User wants emails, delegating to gmail_expert"
worker → gmail_expert (ses_001): {"action": "read_unread", "data": {"limit": 5}}
gmail_expert operates browser, reads emails
gmail_expert → worker (ses_001): {"emails": [...], reasoning: "Found 3 unread emails"}
worker processes, formats for human
worker → human (ses_001): "You have 3 unread emails: ..."
REPL displays response
```

### Expert detects urgent email
```
gmail_expert creates ses_003
gmail_expert → worker (ses_003): {"action": "new_email", "data": {"email": {...}, "urgent": true}}
worker → human (ses_003): "Urgent email from investor@acme.com: ..."
REPL prints: --- NEW SESSION ses_003: gmail_expert — urgent email from investor ---
```
