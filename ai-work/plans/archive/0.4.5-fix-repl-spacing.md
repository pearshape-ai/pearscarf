# Plan 0.4.5: Fix REPL Response Text Spacing
**Date:** 2026-03-01
**Status:** Draft

## Context

Worker responses render with random horizontal offsets. Every line starts at a different column position instead of being flush left. The v0.4.4 indentation fix didn't work because there are two bugs stacking.

## Root Cause

Two issues:

### 1. Raw mode newlines (`terminal.py`)

`TerminalUI.print_above()` is called from the poll thread while `read_line()` has the terminal in raw mode. In raw mode, `\n` (line feed) only moves the cursor **down** — it does NOT return the cursor to column 0. You need `\r\n` (carriage return + line feed) for that.

So when `print_above()` writes multi-line text like:
```
[ses_001] worker > Line one\nLine two\nLine three
```

Each `\n` drops the cursor down but leaves it at whatever column the previous line ended at. This creates the erratic horizontal offsets:
```
[ses_001] worker > Line one
                            Line two
                                     Line three
```

### 2. Unwanted indentation padding (`repl.py`)

The current code pads continuation lines with spaces to match the prefix width:
```python
indent = " " * _visible_len(prefix)  # e.g. 19 spaces
formatted += "\n" + indent + continuation
```

The spec says continuation lines should be flush left at column 0, no padding.

## Fix

### `terminal.py` — Convert `\n` to `\r\n` in `_write()`

All output through `_write()` should replace bare `\n` with `\r\n` so text renders correctly regardless of terminal mode.

```python
def _write(self, text: str) -> None:
    # In raw mode, \n doesn't return to column 0. Use \r\n.
    text = text.replace("\n", "\r\n")
    sys.stdout.write(text)
    sys.stdout.flush()
```

### `repl.py` — Remove continuation indentation

First line gets the `[session] agent >` prefix. Continuation lines start at column 0 with no padding.

```python
lines = content.split("\n")
formatted = prefix + lines[0]
for continuation in lines[1:]:
    formatted += "\n" + continuation
```
