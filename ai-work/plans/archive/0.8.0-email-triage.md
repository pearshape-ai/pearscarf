# Plan 0.8.0: Email Triage
**Date:** 2026-03-02
**Status:** Draft

## Context

ChromaDB is in place (v0.7.0). The Indexer processes all records into the knowledge graph. This version adds triage — the worker classifies incoming emails as relevant or noise before the Indexer processes them. Auto-classifies when confident, asks the human when uncertain. Human responses provide context that enriches the knowledge graph.

## Database Schema Changes (`db.py`)

Add three columns to records (ALTER TABLE with try/except for each):

```sql
ALTER TABLE records ADD COLUMN classification TEXT;           -- "relevant", "noise", "pending"
ALTER TABLE records ADD COLUMN classification_reason TEXT;    -- worker's reasoning or human's words
ALTER TABLE records ADD COLUMN human_context TEXT;            -- additional context from human
```

New records default to `NULL` classification. The gmail expert's `save_email` already inserts records — they'll have `classification = NULL` which we treat as "pending" (unclassified).

## Worker Changes (`agents/worker.py`)

### New tools

**`classify_record`** — Stores a classification on a record:

```python
class ClassifyRecordTool(BaseTool):
    name = "classify_record"
    input_schema = {
        "properties": {
            "record_id": {"type": "string"},
            "classification": {"type": "string", "enum": ["relevant", "noise"]},
            "reason": {"type": "string"},
            "human_context": {"type": "string"},
        },
        "required": ["record_id", "classification", "reason"],
    }
```

**`search_entities`** — Search the knowledge graph:

```python
class SearchEntitiesTool(BaseTool):
    name = "search_entities"
    input_schema = {
        "properties": {
            "query": {"type": "string", "description": "Name, email, or domain to search for"},
            "entity_type": {"type": "string", "description": "Optional: 'person' or 'company'"},
        },
        "required": ["query"],
    }
```

### System prompt update

Add triage instructions to `WORKER_SYSTEM_PROMPT`:

```
Email Triage:
When you receive an email from the gmail_expert, classify it before presenting to the human:

1. Use search_entities to check if the sender is a known entity in the graph.
2. If sender is a known entity → auto-classify as "relevant" using classify_record.
3. If the email has obvious noise signals (no-reply address, unsubscribe, promotional) → auto-classify as "noise".
4. If uncertain → present the email to the human and ask "Is this relevant and why?"
5. When the human responds, use classify_record with their reasoning.

Always send the classification result to the human:
- Relevant: "✓ Relevant: Email from X 'Subject' — reason"
- Noise: "✗ Noise: Email from X 'Subject' — reason"
- If the human overrides a classification, reclassify with classify_record.
```

## Indexer Changes (`indexer.py`)

Change the polling query to only process classified records:

```python
# Before:
"SELECT ... FROM records WHERE indexed = 0 ORDER BY created_at"

# After:
"SELECT ... FROM records WHERE indexed = 0 AND classification = 'relevant' ORDER BY created_at"
```

Noise records are never indexed. Unclassified (NULL) records wait for triage.

## Triage Flow

### Auto-classified (confident)

```
1. Gmail expert reads email → saves as email_005
2. Gmail expert replies to worker with email content + record_id
3. Worker calls search_entities("michael@acmecorp.com")
4. Match found → person_001 (Michael Chen, Acme Corp)
5. Worker calls classify_record("email_005", "relevant", "known entity: Michael Chen at Acme Corp")
6. Worker sends to human: "✓ Relevant: Email from michael@acmecorp.com 'Q3 Invoice' — known entity (Acme Corp)"
7. Indexer picks up email_005 (classified + unindexed) → processes
```

### Uncertain — asks human

```
1. Gmail expert reads email → saves as email_006
2. Gmail expert replies to worker
3. Worker calls search_entities("sarah@unknownco.com") → no match
4. No obvious noise signals
5. Worker sends to human: "New email from sarah@unknownco.com — Subject: 'Intro from mutual friend' — Is this relevant and why?"
6. Human responds: "yes, Alex referred them. Sarah is exploring a partnership."
7. Worker calls classify_record("email_006", "relevant", "Alex referral, potential partnership", human_context="Sarah exploring partnership, referred by Alex")
8. Worker sends to human: "✓ Classified as relevant. Stored context."
9. Indexer picks up email_006 → processes (human_context enriches extraction)
```

### Override

```
1. Worker auto-classifies email_007 as noise
2. Worker sends: "✗ Noise: Email from deals@vendorx.com 'Renewal notice'"
3. Human responds: "no that's relevant, VendorX is our hosting provider"
4. Worker calls classify_record("email_007", "relevant", "hosting provider", human_context="VendorX is hosting provider")
5. Worker sends: "✓ Reclassified as relevant. Stored: VendorX → hosting provider."
```

## Indexer: human_context enrichment

When the Indexer processes a record that has `human_context`, it appends it to the content in the extraction prompt:

```
Record (email_006):
From: sarah@unknownco.com
Subject: Intro from mutual friend
Body: Hi Gev, Alex suggested I reach out...

Additional context from human:
Sarah exploring partnership, referred by Alex
```

This makes the LLM extraction richer — it'll create entities and relationships the email text alone might not surface.
