## Implementation Steps

### 1. Add Gmail MCP config variables (`config.py`)

Add new env vars alongside existing ones:

```python
GMAIL_CLIENT_ID = os.getenv("GMAIL_CLIENT_ID", "")
GMAIL_CLIENT_SECRET = os.getenv("GMAIL_CLIENT_SECRET", "")
GMAIL_REFRESH_TOKEN = os.getenv("GMAIL_REFRESH_TOKEN", "")
GMAIL_POLL_INTERVAL = int(os.getenv("GMAIL_POLL_INTERVAL", "300"))
```

### 2. Add MCP client to Gmail expert (`experts/gmail.py`)

#### 2a. MCP Client Class

Add a `GmailMCPClient` class that wraps the Gmail MCP server connection:

- Uses an existing open-source Gmail MCP server (e.g. `gmail-mcp`) as a subprocess
- Connects via stdio transport using the MCP client SDK
- Provides methods: `list_unread()`, `read_email(id)`, `search(query)`, `mark_as_read(id)`
- Handles OAuth token refresh internally
- Raises clear errors on auth failure

#### 2b. Transport Selection

At expert startup, check for MCP credentials:

```python
def _select_transport(self):
    if config.GMAIL_CLIENT_ID and config.GMAIL_CLIENT_SECRET and config.GMAIL_REFRESH_TOKEN:
        try:
            self._mcp = GmailMCPClient()
            log.write("gmail_expert", "--", "action", "MCP transport active (OAuth configured)")
            return "mcp"
        except Exception as e:
            log.write("gmail_expert", "--", "error", f"MCP init failed: {e}, falling back to browser")
    log.write("gmail_expert", "--", "action", "Browser transport active (no MCP credentials)")
    return "browser"
```

Transport is selected once at startup, not per-request. MCP is default when credentials are present.

#### 2c. MCP-backed Tool Implementations

Add MCP versions of the core tools. Each tool checks `self._transport`:

- `gmail_get_unread`: MCP → `list_unread()`, browser → existing Playwright flow
- `gmail_read_latest`: MCP → `read_email(id)`, browser → existing flow
- `gmail_mark_as_read`: MCP → `mark_as_read(id)`, browser → existing flow
- New tool `gmail_search`: MCP only (no browser equivalent needed)

The existing `save_email` tool is transport-agnostic — it just takes parsed email data and persists it. No changes needed.

#### 2d. Fallback Behavior

If MCP auth fails at runtime (e.g. token expired mid-session):
- Log error
- Do NOT auto-fallback to browser (would require separate browser session)
- Surface error to worker: `"OAuth token expired. Re-authenticate with: pearscaff gmail --auth"`

### 3. Email Polling Loop (`experts/gmail.py`)

#### 3a. Polling Function

Add `start_polling(bus, store)` method to the Gmail expert module:

```python
def start_polling(bus, store, mcp_client, interval):
    """Background thread that polls for new unread emails."""
    while True:
        try:
            unread = mcp_client.list_unread()
            for email in unread:
                existing = store.get_email_by_message_id(email["message_id"])
                if existing:
                    continue  # Already known

                # Save to SOR
                record_id = store.save_email(
                    message_id=email["message_id"],
                    sender=email["sender"],
                    recipient=email["recipient"],
                    subject=email["subject"],
                    body=email["body"],
                    received_at=email["received_at"],
                    raw=email["raw"],
                )
                if not record_id:
                    continue  # Duplicate (race condition safety)

                # Create session + notify worker
                session_id = db.create_session(
                    initiated_by="gmail_expert",
                    summary=f"New email from {email['sender']}"
                )
                bus.send(
                    session_id,
                    from_agent="gmail_expert",
                    to_agent="worker",
                    content=f"New email from {email['sender']}\n"
                            f"Subject: \"{email['subject']}\"\n"
                            f"Record: {record_id}\n\n"
                            f"Is this relevant and why?"
                )
                log.write("gmail_expert", session_id, "action",
                           f"Poll: new email {record_id} from {email['sender']}")
        except Exception as e:
            log.write("gmail_expert", "--", "error", f"Email poll failed: {e}")
            # Surface to human via bus
            _notify_poll_error(bus, str(e))

        time.sleep(interval)
```

#### 3b. Error Notification

Poll errors are sent to the human via the bus on a dedicated error session (or the most recent active session). Format matches spec:

```
worker > ⚠ Email poll failed: OAuth token expired. Re-authenticate with: pearscaff gmail --auth
worker > ⚠ Email poll failed: MCP server unreachable. Retrying in 60s.
```

#### 3c. Threading

Polling runs in a daemon thread started by the CLI command. It shares the same `MessageBus`, `Store`, and `MCP client` instances. Thread safety is already handled by SQLite WAL mode and the existing `threading.Lock()` in logging.

#### 4b. Add `pearscaff gmail --auth` command

New Click command under the `expert` group:

```python
@expert.command("gmail")
@click.option("--auth", is_flag=True, help="Run Gmail OAuth flow")
@click.option("--login", is_flag=True, help="Browser login (existing)")
def gmail_cmd(auth, login):
    if auth:
        run_oauth_flow()  # Opens browser for Google OAuth, stores refresh token
    elif login:
        login()  # Existing headless browser login
    else:
        # Standalone expert mode (existing)
        ...
```

#### 4c. OAuth Flow

`run_oauth_flow()`:
1. Start local HTTP server on `localhost:8080`
2. Open browser to Google OAuth consent URL with Gmail read-only scope
3. Receive callback with auth code
4. Exchange for refresh token
5. Print refresh token and instruct user to add to `.env`

## What Stays the Same

- All existing browser tools — no modifications
- `save_email()` — already transport-agnostic
- System of Record schema — no migrations
- Triage flow — worker classifies same as before
- Indexer pipeline — unchanged
- Session/message architecture — unchanged
- `pearscaff expert gmail --login` — still works for browser sessions

## What NOT to Build

- No Pub/Sub or webhook-based notifications
- No send/reply capabilities via MCP (read-only)
- No automatic switching from browser to MCP — both coexist
- No changes to Retriever or Indexer

## Dependencies

Likely new pip dependencies:
- `mcp` — MCP client SDK (for connecting to Gmail MCP server)
- `google-auth-oauthlib` — for the OAuth flow (or handled by the MCP server)

The specific Gmail MCP server to use should be evaluated at implementation time. Candidates: `gmail-mcp`, `google-workspace-mcp`, or similar. Key requirements: OAuth2 support, list/read/search/mark-as-read operations.
