# Plan 0.9.2: Gmail Session Expiry Detection
**Date:** 2026-03-03
**Status:** Draft

## Context

When the Gmail session expires (cookies in `storage_state.json` become stale), the gmail expert silently fails. It navigates to Gmail, gets redirected to `accounts.google.com/signin`, burns through all its turns trying to work with a login page, hits "Max turns reached", and never sends a reply. The worker never gets a response, the human never sees anything. The system appears dead.

This happened in production — `ps discord` showed no activity, no logs, nothing. Root cause was expired Google session.

## Root Cause

The Gmail-specific tools (`gmail_get_unread`, `gmail_read_latest`) navigate to Gmail and then try to find inbox elements. When the session is expired, the browser gets redirected to `accounts.google.com`. The tools don't check for this — they just report "no elements found" and the LLM keeps trying different approaches until max turns.

## Fix

Two changes:

### 1. Add a session check helper

A function that checks if the current page URL indicates a login redirect:

```python
def _check_session(page) -> str | None:
    """Check if the browser was redirected to a login page.
    Returns an error message if session expired, None if OK."""
    if "accounts.google.com" in page.url:
        return "Gmail session expired. Run 'pearscaff expert gmail --login' to re-authenticate."
    return None
```

### 2. Add the check to Gmail tools

The three Gmail-specific tools (`gmail_get_unread`, `gmail_read_latest`, `gmail_mark_as_read`) call `_check_session()` after navigating. If expired, return the error immediately instead of trying to parse the page.

### 3. Add session check to system prompt

Tell the gmail expert: if a tool returns a session-expired error, immediately reply with that error so the worker can inform the human. Don't try to recover.
