# Plan 0.5.0: System of Record — Initial
**Date:** 2026-03-01
**Status:** Draft

## Context

Worker-expert communication is working. Gmail expert reads emails via headless browser and replies with unstructured text. This version adds persistent structured storage — the System of Record (SOR). Each expert owns writing to its domain tables. The worker reads from storage for context.

Currently, email data flows through the system as unstructured text in messages. After this change, emails are also stored as structured records in the database, enabling deduplication, lookups, and future indexing.

## Database Schema

Added to `init_db()` in `db.py`:

```sql
CREATE TABLE IF NOT EXISTS records (
    id TEXT PRIMARY KEY,
    type TEXT NOT NULL,
    source TEXT NOT NULL,
    created_at TEXT NOT NULL,
    raw TEXT
);

CREATE TABLE IF NOT EXISTS emails (
    record_id TEXT PRIMARY KEY REFERENCES records(id),
    message_id TEXT UNIQUE,
    sender TEXT,
    recipient TEXT,
    subject TEXT,
    body TEXT,
    received_at TEXT
);

CREATE INDEX IF NOT EXISTS idx_emails_message_id ON emails(message_id);
CREATE INDEX IF NOT EXISTS idx_records_type ON records(type);
```

## Gmail Expert Changes (`experts/gmail.py`)

### New tool: `save_email`

The gmail expert gets a `save_email` tool to persist emails to the SOR:

```python
class SaveEmailTool(BaseTool):
    name = "save_email"
    description = "Save an email to the system of record for future reference."
    input_schema = {
        "type": "object",
        "properties": {
            "message_id": {"type": "string", "description": "Gmail's unique message ID (from URL or headers)"},
            "sender": {"type": "string"},
            "subject": {"type": "string"},
            "body": {"type": "string"},
            "received_at": {"type": "string", "description": "Date the email was received"},
        },
        "required": ["sender", "subject", "body"],
    }
```

### System prompt update

Add to `GMAIL_SYSTEM_PROMPT`:
```
After reading an email, ALWAYS save it using the save_email tool before replying.
Include the record_id from save_email in your reply so the worker can reference it.
If save_email returns None (duplicate), skip and note the existing record.
```

## Worker Changes (`agents/worker.py`)

### New tool: `lookup_email`

The worker gets a read-only tool to look up stored emails:

```python
class LookupEmailTool(BaseTool):
    name = "lookup_email"
    description = "Look up a previously stored email by its record ID."
    input_schema = {
        "type": "object",
        "properties": {
            "record_id": {"type": "string", "description": "The email record ID, e.g. 'email_001'"},
        },
        "required": ["record_id"],
    }
```

### System prompt update

Add to `WORKER_SYSTEM_PROMPT`:
```
Emails read by the gmail_expert are stored in the system of record with a record_id.
You can look up previously stored emails using the lookup_email tool.
```

## Data Flow After Change

    ↓

    ↓
