### Design decisions

- **Single collection**: `records` — all record types in one collection, filterable by `type` metadata.
- **Lazy init**: ChromaDB client and sentence-transformers model load on first use, not at import time. This avoids slowing down commands that don't need vector search (like `--version`).
- **Upsert**: Uses `collection.upsert()` so re-indexing the same record is safe.
- **Persistence**: `CHROMA_PATH` defaults to `chroma_data/` alongside the SQLite database.

## `indexer.py` Update

Add Step 2 to `_process_record()` — embed into ChromaDB after graph extraction:

```python
def _process_record(self, record: dict) -> None:
    record_id = record["id"]

    # Step 1: Extract entities, relationships, facts (existing)
    ...

    # Step 2: Embed into ChromaDB (new)
    content = self._build_content(record)
    metadata = {
        "type": record["type"],
        "source": record["source"],
        "created_at": record["created_at"],
    }
    # Add email-specific metadata if available
    if record["type"] == "email":
        email = store.get_email(record_id)
        if email:
            metadata["sender"] = email.get("sender", "")
            metadata["subject"] = email.get("subject", "")

    vectorstore.add_record(record_id, content, metadata)
    log.write("indexer", "--", "action", f"embedded {record_id} in ChromaDB")

    # Mark indexed
    self._mark_indexed(record_id)
```

The `_build_content()` method already exists and builds the email text (From, Subject, Body). It's reused for both the LLM extraction prompt and the embedding.
